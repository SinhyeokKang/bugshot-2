description: 직전에 잡은 버그/회귀를 POSTMORTEM.md에 회고 항목으로 추가. 코드 안 고침.
---

회귀·버그를 잡아 고치고 검증까지 마쳤을 때, "왜 틀렸나 → 다음에 어떻게 막나"를 `POSTMORTEM.md`에 한 항목으로 남기는 스킬. 6축 평가에서 최저였던 **실패 사후분석(FAIL)** 축을 메우는 회로다 — 자동 복구(e2e 루프 등)는 그 자리만 메우지, 같은 함정 재발을 막지 못한다.

**리포트가 아니라 기록 스킬**이다. 코드(`src/`)는 건드리지 않고 `POSTMORTEM.md`만 추가한다. 빌드·커밋 안 함(커밋은 `/push`가 `docs(POSTMORTEM): ...`로).

## 사용

- `/postmortem` — 직전 컨텍스트(방금 고친 버그/회귀)에서 항목을 추론해 작성.
- `/postmortem <설명>` — 대상 명시. 예: `/postmortem logs.html 첨부 누락`.

## 절차

### 1. 사례 확정

직전 대화에서 **무엇이 고쳐졌는지** 확정한다 — 증상, 근본 원인, 손댄 파일. 불확실하면 추측하지 말고 해당 픽스의 diff·커밋을 다시 본다. 픽스가 아직 검증 안 됐으면(테스트 미통과 등) 회고는 이르다 — 먼저 닫으라고 안내하고 중단.

### 2. 비자명성 필터

`POSTMORTEM.md`는 **코드만 읽어선 안 보이는 구조적 함정·재발 패턴**만 담는다. git diff만 봐도 자명한 일회성 오타·단순 누락은 기록하지 않는다(그건 git history가 이미 한다). 남길 가치가 없으면 그렇게 말하고 중단.

판단 기준 — 아래 중 하나면 기록할 가치가 있다:
- 같은 수정을 **여러 곳에 동시에** 해야 했다(복제된 로직·N개 빌더·N개 어댑터).
- 표면 증상과 근본 원인이 **다른 레이어**였다(예: "첨부 안 됨"인데 원인은 본문 빌더 게이트).
- 다음에 같은 류를 막는 **구체적 grep 패턴/전수 대상**이 있다.

### 3. 작성 (POSTMORTEM.md 맨 위 항목으로 추가)

파일 상단 형식 안내를 따른다. `## YYYY-MM-DD — <한 줄 제목>` 헤딩 + 4줄:
- **증상**: 사용자가 관측한 잘못된 동작.
- **근본 원인**: 표면 말고 코드상 진짜 원인.
- **재발 방지**: grep 패턴·전수 대상·테스트 등 구체적 체크.
- **관련**: 손댄 파일·핵심 함수(`file:func` 형태).

기존 항목 위(`---` 구분선 아래 첫 항목)에 끼운다 — 최신이 위. 날짜는 오늘.

### 4. 보고 + 종료

추가한 항목 제목과 핵심 재발방지 한 줄을 보고하고 종료. 빌드·커밋·푸시 안 함.

## 금지 사항

- **코드 수정 금지** — `POSTMORTEM.md`만.
- **자명한 것 기록 금지** — 2단계 필터를 통과 못 하면 중단.
- **빌드·커밋·푸시 금지**.
- **검증 안 된 픽스 회고 금지** — 먼저 픽스를 닫는다.
