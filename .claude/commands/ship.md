작은·외과적 변경 하나를 **`/tdd` → `/implement` → 커밋 → `/code-review` → `/refactor` → `/e2e-write` → (`/guide`) → (`/doc-check`) → `/e2e-run` → `/push` → `/build`** 순서로 자동 오케스트레이션한다. 각 하위 스킬을 순차 호출하고, **단계별 완결 게이트**를 확인해 통과하면 다음으로, 하드 실패나 사용자 결정 지점이면 **즉시 중단하고 현황을 리포트**한다.

이 스킬은 **파이프라인 지휘자**다 — 구현·리뷰·수정의 본체 로직은 각 하위 스킬이 이미 자기완결한다. `/ship`은 순서·게이트·중단만 관장한다.

## 사용

- `/ship <변경 설명>` — 설명을 `/tdd`·`/implement`에 실어 파이프라인 시작.
- `/ship` — 직전 컨텍스트(방금 합의한 변경)에서 대상 자동 판단.

## 적용 범위 (스코프 가드 — 착수 전 판정)

**작은·외과적 변경 전용.** 착수 전 대상을 훑어 아래 중 하나라도 해당하면 **중단하고 수동 흐름(`/feature`부터)을 권한다** — 파이프라인에 태우지 않는다:

- 여러 서브시스템·다중 파일에 걸친 기능 (설계 문서가 필요한 규모)
- 새 권한·OAuth·외부 API·마이그레이션·메시지 union 신설
- `docs/features/<slug>/`가 필요한 신규 기능 (그건 `/feature`→`/feature-review`→… 흐름)

순수 함수 수정, 국소 버그픽스, 단일 컴포넌트 소변경, 표시 포맷 조정 같은 **한 영역 집중 변경**만 대상.

또한 **브랜치 가드**: `git branch --show-current`가 `main`이면 즉시 중단 (push 불가 — `/merge` 안내). `dev` 등 작업 브랜치에서만 진행.

## push 권한

이 파이프라인은 마지막에 **원격 push까지 자동 수행**한다. `/ship` 호출 자체가 그 변경을 push하라는 명시 지시로 간주된다 (평소 "push는 명시 지시 때만" 원칙의 충족 경로). 사용자가 이를 원치 않으면 `/ship`을 호출하지 않는다.

## 파이프라인 단계 + 게이트

각 단계는 하위 스킬을 Skill 도구로 호출한다. **게이트**를 만족해야 다음 단계로 간다. `[영향 게이팅]` 표시 단계는 앞 단계 보고의 플래그가 있을 때만 실행한다.

### 0. 인테이크
- 스코프 가드·브랜치 가드 판정 (위). 실패 시 중단.
- **클린 워크트리 전제**: `git status --porcelain`에 미커밋 변경이 있으면 **파이프라인 시작 전에 먼저 별도 커밋**한다(내용을 파악해 적절한 영문 메시지로 — `/push` step 2와 동일하게 허락 없이). 파이프라인 자체 커밋(feat/fix)과 기존 잔여 변경이 섞이지 않게 격리하려는 것. 단, 미커밋 diff가 **이번 `/ship` 대상 변경과 겹치면**(대상 코드가 이미 손대져 있으면) 커밋하지 말고 **중단+리포트** — `/tdd`가 테스트를 먼저 박는 순서가 깨지므로 사용자에게 처리를 확인한다.
- 변경 설명 확정. TaskCreate로 파이프라인 단계 추적 목록 생성.

### 1. `/tdd`
- 변경을 재현/규정하는 테스트 작성 (regression 또는 interface 모드).
- **게이트**: red 확인됨(또는 `/tdd` 분류표상 스킵 대상 — 이유 한 줄 기록). red를 형성 못 하면(예상외 green) 중단+리포트.

### 2. `/implement`
- 테스트를 green으로 만드는 구현. 4관점 자체 검증 + CTO 게이트 자기완결.
- **게이트**: `pnpm test` 전체 green + `pnpm typecheck` 통과 + 자체 검증 🔴/🟡 전부 해소(2회 루프 내). 미해소 🔴/🟡가 남거나 test/typecheck 실패면 **중단+리포트**.
- **플래그 수집**: 보고의 "가이드 영향"·"e2e 영향"을 기록 (뒤 단계 게이팅에 사용). 문서 영향(새 파일·권한·스킴 등)도 함께 메모.

### 3. 커밋 #1 (feat/fix)
- 구현 + tdd 테스트를 커밋. 영문 메시지. 브랜치가 dev 등 작업 브랜치인지 재확인.

### 4. `/code-review`
- 커밋된 변경분 정적 리뷰. **리포트 전용이라 게이트 없음** — 발견 목록을 5단계로 전달.

### 5. `/refactor` (code-review 발견이 있을 때)
- code-review의 🔴/🟡를 지정해 수정 (회귀 테스트 red→green 규율).
- 발견 0이면 스킵. ⚪만 있으면 스킵(보고).
- **게이트**: 수정 후 test green + typecheck 통과. **회귀 위험 항목**은 `/refactor`가 강행 전 확인을 요구하므로, 그 지점에서 **파이프라인 중단+리포트**(사용자 결정 필요).

### 6. 커밋 #2 (refactor)
- 5단계가 코드를 바꿨을 때만 커밋. 안 바꿨으면 스킵.

### 7. `/e2e-write` [영향 게이팅]
- 2단계 "e2e 영향" 플래그가 있을 때만. 없으면 스킵(사유 보고).
- 시나리오를 spec으로 변환, build:e2e + 실행-수정 루프로 green + 1회 재실행.
- src 수정은 `data-testid` 추가만 허용. 구현 결함이 필요하면 `/e2e-write`가 보고 후 종료 → **파이프라인 중단+리포트**.
- **게이트**: 대상 spec green(연속 통과) 또는 "신규 플로우 없음" 스킵 사유.

### 8. 커밋 #3 (e2e)
- 7단계가 spec·testid를 추가/변경했을 때만 커밋.

### 9. `/guide` [영향 게이팅]
- 2·5단계 "가이드 영향" 플래그가 있을 때만. 없으면 스킵(사유 보고).
- `guide/AUTHORING.md` 규칙대로 ko·en 동시 갱신·검증.
- **게이트**: ko/en 동기 + 검증 통과.

### 10. `/doc-check` [영향 게이팅]
- 2단계 문서 영향 메모(새 파일·디렉터리·권한·스킴·컨벤션·워크플로우 변경 등)가 있을 때만. 없으면 스킵(사유 보고).
- 8개 저장소 문서 양방향 대조 → 항목별 수정.
- **게이트**: 미해소 stale 없음. 사용자 결정이 필요한 stale이 나오면 **중단+리포트**.
- **주의 — push 신선도와 층위가 다르다(중복 아님)**: `/doc-check`은 diff 무관 전수 대조(깊고 무거움, 그래서 게이팅). 13단계 `/push`는 별개로 **이번 push diff에 걸린 문서만** 훑는 얕은 2차 안전망을 **항상** 돈다. doc-check을 스킵해도 push의 자체 신선도 검사는 도는 게 정상(CLAUDE.md "문서 신선도" 참조). 같은 걸 두 번 보는 게 아니라 층이 다르다.

### 11. 커밋 #4 (docs)
- 9·10단계가 guide/문서를 바꿨을 때만 커밋(`docs(...)` 스코프).

### 12. `/e2e-run` (최종 전수 게이트)
- **모든 커밋 이후**에 실행 — 그래야 green 시 `e2e/.last-green`이 최종 HEAD를 가리켜 다음 `/push`의 e2e 게이트가 재실행 없이 통과한다.
- build:e2e + 전체 test:e2e.
- **게이트**: 전체 green. 클린 트리면 `.last-green`에 HEAD 기록. **빨간불이면 push로 진행하지 않고 즉시 중단+리포트**(fix는 사용자가 `/e2e-write`·`/implement`로).

### 13. `/push`
- dev push + 문서 신선도 검사 + e2e 게이트(`.last-green`==HEAD면 스킵). 12단계에서 트리·HEAD를 맞췄으므로 통상 재실행 없이 통과.
- **게이트**: 신선도 검사 통과 + push 성공. 신선도 미충족(diff에 걸린 문서 stale)·push 거부 시 **중단+리포트**.

### 14. `/build`
- `pnpm build` + 테스트 체크리스트(최종 산출물 검증).
- **게이트**: 빌드 성공. 실패면 리포트(이미 push된 상태이므로 후속 픽스 필요를 명시).

### 15. 최종 리포트
- 단계별 통과/스킵/중단 요약 + 커밋 해시들 + push 결과 + 남은 수동 항목.

## 중단 처리 (즉시 중단+리포트)

어느 단계든 게이트 실패나 사용자 결정 지점에서 파이프라인을 멈추고 보고한다:

```
🛑 중단: <단계> 게이트 실패
사유: <한 줄>
현재 상태: 커밋 <해시들> / 미커밋 <있으면>
남은 단계: <목록>
재개: <사용자가 무엇을 결정/수정하면 이어갈 수 있는지>
```

중단 후 **임의로 우회·강행하지 않는다**. 사용자 지시를 기다린다. (하위 스킬이 자체 수정 가능한 범위 — `/implement`·`/refactor`의 자가수정 루프 — 는 그 스킬 안에서 처리되고, 그걸 넘어선 하드 실패만 파이프라인이 중단한다.)

## 다른 스킬과의 분리

- 개별 스킬(`/tdd`·`/implement`·…) — 단계 하나씩 수동 진행.
- `/ship` ← 여기. **작은 변경의 전 단계 자동 오케스트레이션 + 게이트 + push**.
- `/merge`·`/deploy` — main 반영·배포. `/ship`은 dev push까지만 (main은 사용자가 `/merge`).

## 금지 사항

- **큰·다영역 변경에 사용 금지** — 스코프 가드에 걸리면 수동 흐름 권하고 종료.
- **게이트 실패 우회·강행 금지** — 즉시 중단+리포트, 사용자 결정 대기.
- **main에서 실행 금지** — 브랜치 가드.
- **하위 스킬의 금지 사항 무력화 금지** — 각 스킬의 제약(빌드 범위·수정 범위 등)을 파이프라인이 넘어서지 않는다.
- **단계 임의 생략 금지** — `[영향 게이팅]` 단계만 플래그 부재 시 스킵(사유 보고). 나머지는 순서대로.
