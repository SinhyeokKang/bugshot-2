# 30s Replay

## 배경

BugShot의 현재 비디오 모드는 "시작 → 녹화 → 정지"의 수동 흐름이다. 사용자가 버그를 **발견한 후** 녹화를 시작하면 이미 버그 발생 순간은 지나가 있다. 재현해야 하고, 재현이 안 되면 영상이 쓸모없다.

"항상 녹화 중, 필요할 때 직전 30초를 잡아냄"이 이상적이다. BugShot은 Jira/GitHub/Linear/Notion에 **MP4 파일을 직접 첨부**해야 하므로, 클라이언트에서 빠르게 인코딩까지 완료해야 한다.

## 전제 및 위험 (제품 결정)

이 기능은 기존 모드와 달리 3가지 무거운 전제 위에 선다. 채택 시 이 비용을 함께 받아들이는 것이다.

1. **"모든 사이트" host permission**: Side Panel은 `activeTab` grant를 받지 못해(Chromium #40916430), 상시 캡처에는 `https://*/*` + `http://*/*` 권한을 런타임 요청해야 한다. 이는 "방문하는 모든 웹사이트의 데이터 읽기/변경" 동의를 의미하므로, **기본 OFF + 설정에서 명시적 opt-in**으로만 활성화한다. 권한 범위와 이유를 설정 UI에서 안내한다.
2. **WebCodecs + mp4-muxer 신규 의존성**: 클라이언트 H.264 인코딩은 검증되지 않은 신규 파이프라인이다. WebCodecs `VideoEncoder` 또는 H.264 코덱이 없는 환경(일부 Linux/구형 Chrome)에서는 **기능 전체가 동작 불가** → 이 경우 30s Replay 버튼을 비활성화한다. ⚠️ **구현 착수 전 WebCodecs PoC 선행 검증 필수** (현재 문서상 가정 단계).
3. **captureVisibleTab rate limit**: Chrome은 초당 2회로 제한. 500ms 간격은 정확히 한계선이라 이전 호출 미완료 시 틱 스킵 가드가 전제다.

## 목표

1. 사이드패널이 열려 있고 **이슈 작성 대기(idle) 상태인 동안** 현재 탭의 스크린을 캡처한다. (드래프팅·프리뷰·수동 녹화 중에는 캡처를 멈춘다 — 시나리오 8 참조. "상시"가 아니라 "idle 한정"이 의도된 동작)
2. 사용자가 "30s Replay" 클릭 시 직전 30초를 H.264 MP4로 인코딩하여 **기준 기기(Apple M1, 60프레임, 1280px 폭)에서 3초 이내에** 비디오 모드 드래프팅에 진입한다.
3. 기존 콘솔/네트워크 로그 첨부 흐름을 그대로 재사용한다.
4. 기존 수동 비디오 녹화 모드는 유지한다 (공존).

## 비목표 (Non-goals)

- rrweb DOM 녹화/재생: 검증 결과 픽셀 변환 비용이 비현실적. 제외.
- 서버사이드 인코딩: 프라이버시·인프라 이유로 제외.
- 오디오 캡처: 탭 오디오 불필요.
- 30초 초과 버퍼: 메모리 제약상 30초로 고정.
- 버퍼 지속 시간 사용자 설정: V1에서 지원하지 않는다. 30초 고정.
- rrweb 기반 인터랙티브 리플레이 뷰어: 스코프 외.
- file:// 프로토콜 페이지: `optional_host_permissions`에 http/https만 선언되어 있으므로 V1에서 미지원.

## 사용자 시나리오

### 시나리오 1: 일반 사용

1. 사이드패널이 열려 있고 프레임 캡처가 진행 중. 최소 10장(5초) 확보 전까지 30s Replay 버튼 비활성.
2. 5초 경과 후 버튼 활성화. 사용자가 웹 페이지에서 버그를 발견한다.
3. "30s Replay" 버튼 클릭.
4. 1-2초간 인코딩 진행 표시 → MP4 생성 완료.
5. 비디오 모드 드래프팅 화면 진입 (썸네일 + 타이틀 + 마크다운 에디터).
6. 콘솔/네트워크 로그가 자동 첨부 (기존 로직 동일).
7. 이후 흐름은 기존 비디오 모드와 동일 (프리뷰 → 플랫폼 제출).

### 시나리오 2: 버퍼 부족 (최소 프레임 미확보)

1. 사이드패널을 연 직후 (5초 미만) 30s Replay 버튼이 비활성 상태.
2. 10장(5초) 확보 시 버튼 자동 활성화.

### 시나리오 3: 탭 전환

1. 사용자가 다른 탭으로 전환.
2. 바운드 탭이 비활성이면 캡처 일시 중지. 버퍼의 기존 프레임은 유지된다.
3. 바운드 탭으로 돌아오면 캡처 재개.
4. 비활성 시간만큼 실제 캡처된 영상은 30초보다 짧을 수 있다 (예: 10초 비활성 → 20초 분량).

### 시나리오 4: 탭 네비게이션

1. 사용자가 같은 탭에서 다른 URL로 이동한다.
2. 버퍼를 클리어하지 않는다 — 이전 페이지 프레임과 새 페이지 프레임이 연속된다.
3. 버그 재현 과정에서 페이지 전환이 포함될 수 있으므로 의도된 동작이다.
4. 단, 미지원 스킴(file://, chromewebstore 등)으로 이동하면 캡처가 끊겨 그 구간은 버퍼에 공백으로 남는다 (재개 시 프레임 간격이 벌어짐 → 인코딩 시 duration cap으로 처리, design 참조).

### 시나리오 5: 수동 비디오 녹화 중 30s Replay

1. 사용자가 수동 비디오 녹화 중(`phase: "recording"`)에 "30s Replay" 클릭.
2. 30s Replay 버튼 비활성 — 수동 녹화와 동시 사용 불가.
3. 수동 녹화 완료 → `phase: "drafting"` 진입. 캡처 루프는 idle이 아니므로 중지 상태.
4. 사용자가 드래프트를 완료/취소해 idle로 복귀하면 캡처 루프 재개, 5초 대기 후 30s Replay 사용 가능.

### 시나리오 6: 인코딩 실패

1. WebCodecs 코덱 지원 불가 또는 메모리 부족 등으로 인코딩이 실패한다.
2. toast 알림으로 에러 메시지 표시.
3. 기존 버퍼의 프레임은 보존된다 (copy-then-clear 전략: 인코딩 성공 시에만 버퍼 클리어).
4. 캡처 루프 재개, 사용자는 즉시 다시 시도 가능. 기존 모드 정상 동작.

### 시나리오 7: 사이드패널 닫힘/재열림

1. 사용자가 사이드패널을 닫는다.
2. `use30sReplay` hook 언마운트 → 캡처 루프 중지 + 버퍼 클리어.
3. 사이드패널을 다시 열면 캡처 루프가 0부터 재시작. 최소 5초 대기 후 30s Replay 사용 가능.

### 시나리오 8: 30s Replay 완료 후

1. 사용자가 30s Replay 클릭 → 인코딩 완료 → 드래프팅 진입.
2. 캡처 루프는 idle 상태에서만 동작하므로, 드래프팅/프리뷰 중에는 캡처가 중지된다.
3. 사용자가 드래프트를 완료/취소해 idle로 복귀하면 캡처 루프 재개, 5초 대기 후 다시 사용 가능.

## 성공 기준

- [ ] 설정 ON + 사이드패널 열림(idle) → 500ms 간격 프레임 캡처 자동 시작
- [ ] "30s Replay" 클릭 → 기준 기기(M1, 60프레임, 1280px)에서 p95 ≤ 3초로 MP4 blob 생성 + 드래프팅 진입
- [ ] 생성된 MP4가 Jira/GitHub/Linear/Notion 이슈 첨부 시 정상 재생 가능
- [ ] 메모리 사용량: 프레임 버퍼 ≤ 15MB (레티나 기준 — quality 65 + 캡처 시점 리사이즈 적용 후 산정)
- [ ] 캡처 루프 동작 중 캡처로 인한 추가 CPU 사용률 10% 이하 (long task >50ms 발생 0건)
- [ ] 30s Replay로 생성된 이슈가 IndexedDB에 정상 영속화됨 (confirmDraft 경로)
- [ ] 기존 수동 비디오 녹화 모드 정상 동작 (회귀 없음)
- [ ] 콘솔/네트워크 로그 첨부 기존 흐름과 동일하게 동작

### Graceful degradation 기준

- [ ] 권한 거부 시: Switch OFF 복귀 + toast 안내, 캡처 미시작
- [ ] WebCodecs/H.264 미지원 환경: 30s Replay 버튼 비활성 (클릭 불가)
- [ ] 인코딩 실패 시: toast 에러 + 버퍼 보존(copy-then-clear) + 즉시 재시도 가능
- [ ] 권한 외부 철회(chrome://extensions) 후 패널 재열기: 자동 OFF + toast
