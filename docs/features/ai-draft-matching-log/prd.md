# AI 초안 — 성공(200) 로그 매칭 증거

## 배경

AI 초안은 지금 네트워크 로그 중 **에러만**(`status >= 400 || phase==="error"`, `buildLogSummary.ts:45`) 후보로 담아 모델에 보낸다. 그런데 버그의 원인은 200 응답에 있는 경우가 오히려 흔하다:

- **백엔드 데이터 누락** — 200인데 `items: []`, 특정 필드 `null`
- **API 계약 위반** — 스키마 어긋남, 필드 리네임, 타입 불일치
- **프론트 상태 매핑 실패** — 응답은 정상인데 프론트가 다른 형태를 기대(소통 격차에서 자주 발생)

이 클래스는 **에러를 전혀 안 남긴다**(200·콘솔 깨끗·네트워크 정상). 현재의 "에러만 담기"가 구조적으로 실명(失明)인 영역이 정확히 여기다. 이 세 케이스 중 둘은 버그가 **응답 본문 안에** 있어서, 본문 파생 증거 없이는 모델이 원인을 지목할 수 없다.

## 목표

- AI 초안 후보 풀에 **유저·캡처 컨텍스트와 exact-match되는 성공 응답**을 추가한다. 선별은 휴리스틱이 아니라 기존 `requestMatchesQuery`(`@/lib/network-search`)를 재사용한 exact substring 매칭이다.
- 매칭 소스를 유저 텍스트에 국한하지 않고 **콘솔 에러 메시지·액션 로그·선택 요소 selector**까지 확장해, 유저가 식별자를 붙여넣지 않아도 원인 200을 적중시킨다.
- 매칭된 200의 **shape 다이제스트**(키·타입·길이, 값 제외)를 rich 프롬프트에 실어, 모델이 "정상으로 보이는 200"과 "실제 깨진 200"을 구분해 원인 로그를 `logRefs`로 지목하게 한다.
- 지목된 로그는 기존 `renderLogRefBlocks` 경로로 **원문 그대로**(마스킹·절삭된 request/response 본문) 사이드패널 초안에 코드블럭으로 삽입된다 — 추가 삽입 코드 없음.

## 비목표 (Non-goals)

- **timeline-proximity 선별** 도입 안 함. 캡처 순간 근처 200을 담는 방식은 anchor 시점 선정 + API가 아닌 200(애널리틱스·이미지·청크) denoise가 필요해 휴리스틱을 재유입시킨다.
- **compact 스타일 지원 안 함.** 소형 모델은 "이 200이 사실 버그"라는 추론을 못 하고 예산도 좁다. rich 전용.
- **응답 본문 원값을 LLM에 전송 안 함.** LLM에는 shape 다이제스트(키·타입)만. 실제 값은 앱-사이드 삽입(개발자만 확인)에만 존재.
- **매처 로직 변경 안 함.** `requestMatchesQuery`는 그대로 재사용. 새로 만드는 건 유저 입력을 term으로 쪼개는 토크나이저뿐.
- **새 UI 없음.** 후보 선별·프롬프트 구성만 바뀌고 사이드패널 화면·버튼은 그대로.

## 사용자 시나리오

1. 유저가 freeform/video/screenshot 모드로 버그를 캡처하고 로그가 수집된다(콘솔에 `TypeError: ...reading 'orderStatus'`, 네트워크에 `GET /api/orders → 200`이 `{items: []}` 반환).
2. AI 초안 다이얼로그에 `"주문 목록이 안 떠요"`처럼 **순수 한글 한 줄**만 입력하고 생성.
3. 토크나이저가 유저 입력 + **콘솔 에러 메시지**에서 `orderStatus`를, 액션/selector에서 추가 term을 뽑는다.
4. `orderStatus`(또는 다른 term)가 `GET /api/orders`의 응답 본문에 exact-match → 매칭 후보 `[m1] GET /api/orders → 200 · {items:arr[0] order_status:str} (matched "orderStatus")`가 rich 프롬프트에 실린다.
5. 모델이 다이제스트로 "필드는 present인데 유저는 안 보인다 함 → 상태 매핑 가설" 또는 "items 비어있음 → 데이터 누락"을 추론, `logRefs: ["m1"]` 반환 + notes에 가설.
6. 앱이 `m1`을 원본 요청으로 되짚어 request/response 원문을 description 코드블럭으로 삽입. 개발자는 "백엔드가 실제 보낸 값"을 이슈에서 바로 확인한다.

### 엣지 케이스

- **매칭 0**: 유저·캡처 텍스트가 어떤 응답에도 literal로 안 걸림 → 매칭 후보 없음. 에러 후보만으로 기존과 동일 동작(무해·additive).
- **over-match term**: `api` 같은 흔한 term이 8건 초과 매칭 → 그 term은 신호가 아니라 노이즈로 폐기.
- **예산 초과**: `fitDraftContext`가 로그를 트리밍(level ≥ 1)하면 매칭 후보도 함께 사라짐 — 에러 후보·요약과 동일 취급.
- **json이 아닌/omitted 응답**: 다이제스트 불가 → 다이제스트 없이 provenance 줄만 인쇄(매칭 근거는 유지).
- **compact 프로바이더**: 매칭 후보 생성 자체를 건너뜀 → 기존과 완전히 동일.

## 성공 기준

- 콘솔 에러가 특정 필드명을 언급하고 그 필드가 200 응답에 있으면, 유저가 그 필드명을 안 쳐도 해당 200이 매칭 후보로 rich 프롬프트에 인쇄된다(단위 테스트로 검증).
- 다이제스트에 응답 **값**이 절대 포함되지 않는다(키·타입·길이만) — 단위 테스트로 값 부재를 잠근다.
- 매칭 후보가 `m1..` ref로 `candidateRefs`·schema enum·`findCandidate`·`renderLogRefBlocks`에 일관되게 편입된다(위조 ref는 후보 대조에서 걸러지는 기존 방어선 유지).
- compact 스타일과 요약 부재 시 매칭 후보가 빈 배열이다.
- `pnpm test` 통과, `pnpm typecheck` 통과.
