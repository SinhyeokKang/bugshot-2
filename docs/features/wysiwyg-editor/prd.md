# WYSIWYG Editor for Issue Sections

## 배경

Bugshot의 이슈 작성 필드("발생 현상", "기대 결과", "비고")는 현재 plain text `<Textarea>`를 사용한다. 사용자가 텍스트 서식(Bold, Italic, 리스트 등)을 적용하거나 스크린샷을 본문에 직접 삽입할 수 없어, 풍부하고 구조적인 버그 리포트 작성이 제한된다.

사용자 피드백에서 "스크린샷을 본문에 바로 넣고 싶다", "서식 없이 밋밋하다"는 요청이 반복되고 있고, 이슈 등록 후 플랫폼에서 본문을 재편집하는 비율이 높아 이 시점에 우선순위를 둔다.

마크다운 문법을 입력하면 실시간으로 서식이 적용되는 WYSIWYG 에디터를 도입해, 별도 툴바 없이 Notion/Typora 스타일의 자연스러운 리치 텍스트 작성 경험을 제공한다.

## 목표

1. `renderAs: "paragraph"` 섹션(description, expectedResult, notes)의 plain text Textarea를 Tiptap 기반 WYSIWYG 에디터로 교체한다.
2. 마크다운 문법 입력 시 자동으로 서식이 적용된다 (InputRules 기반, 툴바 없음).
3. 이미지를 드래그앤드롭 또는 클립보드 붙여넣기로 섹션 본문에 인라인 삽입할 수 있다.
4. 내부 저장 포맷은 마크다운 문자열을 유지해 기존 데이터와 하위 호환된다.
5. 4개 플랫폼(Jira, GitHub, Linear, Notion) 이슈 제출 시 서식과 인라인 이미지가 정확히 반영된다.

## 비목표 (Non-goals)

- **Heading**: 섹션 본문 내 제목은 지원하지 않는다. 이슈 구조는 기존 섹션 분리로 충분하다.
- **Code block**: 인라인 코드(`code`)만 지원. 멀티라인 코드 블록은 이번 스코프에서 제외.
- **Table**: 본문 내 표 작성은 지원하지 않는다.
- **Toolbar/버튼 UI**: 서식 적용은 마크다운 단축키로만. 툴바나 포맷 버튼을 제공하지 않는다.
- **stepsToReproduce 섹션**: `renderAs: "orderedList"`인 "재현 과정" 섹션은 기존 `OrderedListEditor`를 유지한다. 향후 OrderedListEditor를 Tiptap으로 통합하는 것은 별도 검토.
- **디스커버러빌리티 장치**: 온보딩 힌트, 슬래시 커맨드, 포맷 팁 등 서식 기능 안내 UI는 이번 스코프에서 제외.
- **플랫폼별 서식 제한**: 에디터는 모든 지원 서식을 허용하고, 특정 플랫폼에서 미지원되는 서식은 제출 시 silent fallback(plain text)으로 처리. 사전 경고 UI는 제공하지 않는다.
- **마크다운 소스 보기**: 마크다운 원문을 직접 편집하는 토글/모드는 제공하지 않는다.

## 사용자 시나리오

### 시나리오 1: 마크다운 자동 변환으로 서식 적용

1. 사용자가 DraftingPanel에서 "발생 현상" 섹션에 포커스한다.
2. `**중요한 버그**`를 입력하면 마지막 `**` 입력 시점에 자동으로 **중요한 버그**로 변환된다.
3. `*기울임*`, `` `코드` ``, `~~취소선~~` 도 동일하게 자동 변환된다.
4. `[링크 텍스트](https://example.com)` 입력 시 클릭 가능한 링크로 변환된다.
5. 줄 시작에서 `- `(대시+공백) 입력 시 bullet list로 전환, `1. ` 입력 시 numbered list로 전환된다.
6. 변환된 서식은 에디터에 즉시 시각적으로 반영된다 (WYSIWYG).

### 시나리오 2: 이미지 드래그앤드롭

1. 사용자가 데스크탑이나 브라우저에서 이미지 파일을 "기대 결과" 섹션 에디터로 드래그앤드롭한다.
2. 이미지가 blob-db(IndexedDB)에 저장되고, 에디터 내에 인라인으로 표시된다.
3. 이미지는 에디터 폭에 맞게 `max-width: 100%`로 표시된다.
4. 이슈를 GitHub에 제출하면 이미지가 GitHub에 업로드되고 마크다운 본문에 URL로 삽입된다.

### 시나리오 3: 이미지 클립보드 붙여넣기

1. 사용자가 스크린샷을 찍어 클립보드에 복사한다 (Cmd+Shift+4 등).
2. "비고" 섹션 에디터에서 Cmd+V로 붙여넣는다.
3. 시나리오 2와 동일하게 이미지가 저장·표시·제출된다.

### 시나리오 4: AI 드래프트와의 호환

1. 사용자가 AI 드래프트 기능으로 이슈 초안을 생성한다.
2. AI가 생성한 plain text가 에디터에 로드된다 (plain text는 유효한 마크다운).
3. 사용자가 에디터에서 서식을 추가하거나 이미지를 삽입해 보강한다.

### 시나리오 5: 프리뷰/상세 보기

1. 사용자가 PreviewPanel이나 IssueDetailDialog에서 저장된 이슈를 본다.
2. 마크다운 서식(Bold, 리스트 등)이 렌더링되어 표시된다.
3. 인라인 이미지도 blob-db에서 로드되어 표시된다.

### 엣지 케이스

- **빈 섹션**: 기존과 동일하게 "(없음)" 표시.
- **매우 긴 마크다운**: 에디터가 `[field-sizing:content]`로 자동 확장.
- **이미지 삭제**: 에디터에서 이미지 노드를 Backspace로 삭제하면 마크다운에서 참조 제거. 이슈 저장 시 현재 sections 내 `inline:refId` 목록과 비교하여 미참조 blob을 정리하고, 이슈 삭제 시에도 일괄 정리.
- **오프라인/blob-db 실패**: 이미지 저장 실패 시 에디터에 삽입하지 않고 무시.
- **클립보드 복사 시 이미지**: `buildIssueHtml`/`buildIssueMarkdown` 복사 시 `inline:refId` 참조를 data URL로 치환하여 외부 앱 붙여넣기에서 이미지가 깨지지 않도록 한다.
- **기존 plain text 데이터**: plain text는 유효한 마크다운이므로 에디터가 그대로 렌더링. 마이그레이션 불필요.

## 성공 기준

1. `renderAs: "paragraph"`인 3개 섹션에서 Tiptap 에디터가 정상 동작한다.
2. `**`, `*`, `` ` ``, `~~`, `- `, `1. `, `[text](url)` 마크다운 문법이 입력 시 자동 변환된다.
3. 이미지 드래그앤드롭과 Cmd+V 붙여넣기로 인라인 이미지가 삽입된다.
4. 4개 플랫폼(Jira ADF, GitHub MD, Linear MD, Notion Blocks) 제출 시 서식과 이미지가 정확히 반영된다.
5. 프리뷰/상세 다이얼로그에서 마크다운이 렌더링되어 표시된다.
6. 기존 plain text 이슈 데이터가 에디터에서 깨지지 않고 로드된다.
7. 번들 크기 증가가 lazy loading으로 초기 로드에 영향을 주지 않는다.
