# 스타일 편집 코드 뷰 (DevTools식 투트랙)

## 배경

현재 요소 스타일 편집(`StyleEditorPanel`)은 Figma식 큐레이션 폼 단일 트랙이다. `SECTION_PROPS`에 미리 정의한 속성만 섹션별 컨트롤(Select·Text·Quad 등)로 노출한다. 이 방식은 CSS를 모르는 기획자·디자이너에게 적합하지만 두 가지 한계가 있다.

1. **속도**: CSS를 아는 사용자는 크롬 개발자도구의 `element.style {}` 블록에 `padding: 3rem; color: #fff`처럼 raw로 타이핑하는 게 폼 컨트롤을 하나씩 조작하는 것보다 빠르다.
2. **커버리지**: 폼이 큐레이션하지 않은 임의 속성(`cursor`, `content`, `-webkit-line-clamp`, `aspect-ratio` 등)은 아예 편집할 수 없다.

두 사용자 층(기획/디자이너 ↔ 개발자)을 모두 만족시키기 위해 편집 UX를 **폼 / 코드 투트랙**으로 제공한다.

## 목표

- `StyleEditorPanel`의 **스타일 속성 편집 영역만** 폼(현행) / 코드(신규) 두 모드로 스왑하는 탭을 제공한다.
- 두 모드는 같은 `editor-store.styleEdits.inlineStyle`을 공유해 **완전 동기화**된다: 한 모드에서 편집한 값이 즉시 다른 모드에 반영된다.
- 코드 모드는 `styleEdits.inlineStyle`(유저가 실제로 넣은 인라인 오버라이드)을 편집 가능한 raw CSS 텍스트로 노출하고, 타이핑을 파싱해 같은 store에 반영한다.
- 코드 모드에서 **폼에 없는 임의 CSS 속성**을 추가·제거할 수 있다.
- 마지막으로 사용한 편집 모드를 영속화해 다음 진입 시 초기값으로 쓴다(별도 설정 화면 없음).
- 기존 공유 UI — DOM 선택/재선택, 클래스·텍스트 편집, 변경사항 다이얼로그, before/after 비교, 다음 단계 진행 게이트 — 는 **그대로 공유**하고 회귀 없이 동작한다.

## 비목표 (Non-goals)

- **매칭 규칙(`.class {}`) 자체 편집 금지**. 코드 뷰는 요소별 인라인 오버라이드(`element.style {}`)만 편집한다. 페이지 전역 stylesheet를 수정하는 DevTools의 규칙 편집 기능은 스코프 밖이다(이유: design.md "편집 스코프" 참조).
- **매칭 소스 참조 뷰 금지(v1)**. DevTools의 아래쪽 매칭 규칙 리스트를 재현하는 read-only 참조 뷰는 v1 스코프에서 제외한다(코드 뷰 = 편집 textarea만). `propSources`는 prop별 승자 셀렉터 하나뿐이라 정보 밀도가 낮고 캐스케이드 creep의 진입점이다. 필요 시 v2 후속으로 분리한다.
- **DevTools식 전체 캐스케이드 재현 금지**. 모든 매칭 규칙을 소스 위치(`file:line`)·덮어쓰인 선언까지 나열하는 뷰는 만들지 않는다.
- **숏핸드↔롱핸드 정규화 금지**. `inlineStyle`을 롱핸드 그대로 직렬화한다. 폼이 롱핸드로 저장하므로 코드 뷰도 롱핸드로 노출된다(시나리오 B 참조).
- **코드 모드 자체 undo/redo 스택 금지**. 브라우저 네이티브 `textarea` undo만 제공한다.
- **폼 컨트롤 확장 금지**. 코드 뷰가 다루는 임의 속성을 폼에 새 컨트롤로 추가하지 않는다.
- **CSS 문법 자동완성·린팅·신택스 하이라이팅 금지**. 코드 뷰는 순수 `textarea`다.
- **클래스·텍스트 편집 방식 변경 금지**. 두 모드 모두 기존 `ClassEditor`·`TextEditor`를 그대로 쓴다(이들은 CSS 선언이 아니므로 모드 스왑 대상이 아니다).

## 사용자 시나리오

### 주 시나리오 A — 개발자가 코드 모드로 빠르게 편집
1. 요소를 선택하면 styling 화면에 진입한다.
2. 스타일 영역 상단의 세그먼트 토글에서 "코드"를 선택한다(마지막 사용 모드가 코드였다면 이미 선택됨).
3. `element.style {}`에 해당하는 textarea에 `padding: 3rem;\ncursor: pointer;`를 입력한다.
4. 타이핑 즉시 대상 요소에 라이브 반영되고, 변경사항 다이얼로그에도 두 선언이 나타난다.
5. "다음"으로 진행하면 after 스냅샷이 캡처되고 기존 플로우대로 이어진다.

### 주 시나리오 B — 모드 간 왕복 동기화
1. 폼 모드에서 `padding`을 `24px`로 바꾼다(폼은 이 값을 **롱핸드** `padding-top/right/bottom/left`로 저장한다 — `StyleEditorPanel.tsx`의 `QuadProp`).
2. "코드" 토글로 전환하면 textarea에 롱핸드 4줄(`padding-top: 24px;` …)이 이미 들어있다.
3. textarea에서 `border-top-left-radius: 8px;`(롱핸드)를 추가한다.
4. "폼" 토글로 돌아가면 border-radius 컨트롤에 `8px`가 반영돼 있다.

> **숏핸드 입력의 한계(의도된 동작)**: 코드 모드에서 `padding: 24px;`처럼 **숏핸드**로 치면 대상 요소(DOM)에는 적용되지만, 폼 컨트롤은 롱핸드 키(`padding-top` 등)를 읽으므로 폼 모드로 돌아가면 그 컨트롤은 빈 채로 남는다(값 자체는 `inlineStyle`·DOM에 보존). 코드 뷰는 `inlineStyle`을 **롱핸드 그대로** 직렬화하며 숏핸드↔롱핸드 정규화는 하지 않는다(비목표 참조).

### 엣지 케이스
- **폼 미지원 속성 왕복**: 코드 모드에서 `cursor: pointer`를 넣고 폼 모드로 전환하면, `cursor`는 폼에 컨트롤이 없어 **화면에 안 보이지만** store·DOM·변경사항·저장에는 그대로 남는다. 다시 코드 모드로 가면 `cursor: pointer`가 여전히 textarea에 있다. (이 속성을 편집·제거하려면 코드 모드를 쓴다.)
- **잘못된 CSS 입력**: `colr: red`(오타 prop)나 `padding:`(값 없음) 같은 입력은 파서가 관대하게 처리한다 — 값 없는 선언은 무시, 알 수 없는 prop은 store엔 담기되 `setProperty`가 조용히 무시(라이브 반영 없음). 앱은 깨지지 않는다.
- **`!important`**: 코드 모드에서 `color: red !important`를 입력하면 `!important`가 보존돼 적용된다(Tailwind 등 `!important` 페이지 규칙을 오버라이드하려면 필요). 처리 상세는 design.md.
- **속성 제거**: (1) 내가 넣은 오버라이드 제거 = textarea에서 그 줄을 지운다 → inline 맵에서 키 삭제 → 페이지 원래 값 복귀. (2) 페이지 CSS 값 중화 = `padding: 0` / `padding: unset` 등 오버라이드를 명시(인라인은 삭제가 아니라 덮어쓰기만 가능 — DevTools도 동일).
- **첫 진입 기본값**: 영속된 마지막 모드가 없으면 "폼"으로 시작한다(기존 사용자 경험 유지).
- **빈 상태**: 오버라이드가 0개(요소 선택 직후 최빈 케이스)이면 코드 textarea는 완전히 비어 있다. 무엇을 어떤 포맷으로 칠지 어포던스를 주기 위해 placeholder 힌트(예: `padding: 3rem;`)를 노출한다(shadcn `Textarea`의 placeholder).
- **무효 CSS 피드백**: 오타 prop·값 없는 선언은 파서가 조용히 무시하므로, 사용자가 받는 유일한 피드백은 "라이브 프리뷰가 안 바뀜"이다. 별도 인라인 에러·린팅은 제공하지 않는다(비목표). 유효 CSS는 즉시 요소에 반영돼 그 자체가 피드백이 된다 — 대상 사용자(개발자) 기준 수용 가능한 알려진 한계다.

## 성공 기준

- styling 화면에서 폼/코드 토글이 노출되고, 전환 시 스타일 편집 영역만 바뀐다(DOM 네비·클래스·텍스트·푸터·다이얼로그는 불변).
- 코드 모드 textarea 편집이 폼 모드와 동일하게 `styleEdits.inlineStyle`을 갱신하고 대상 요소에 라이브 반영된다.
- 폼→코드→폼 왕복 시 **값 유실이 없다**(폼 미지원 속성·숏핸드 입력 포함). 단 "값 보존"과 "폼 컨트롤 가시성"은 구분한다 — 폼 미지원 속성·숏핸드 값은 `inlineStyle`·DOM·변경사항·저장에 보존되지만 폼 컨트롤에는 표시되지 않을 수 있다(의도된 동작).
- 폼에 없는 임의 속성을 코드 모드로 추가·제거할 수 있고, 변경사항 다이얼로그·before/after·저장(draft)에 반영된다.
- 마지막 사용 모드가 영속되어 다음 styling 진입 시 그 모드로 시작한다.
- inlineStyle ↔ CSS 텍스트 양방향 변환 순수 함수에 단위 테스트가 있고 `pnpm test` 통과.
