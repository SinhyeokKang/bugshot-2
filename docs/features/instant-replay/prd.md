# Instant Replay

## 배경

BugShot의 현재 비디오 모드는 "시작 → 녹화 → 정지"의 수동 흐름이다. 사용자가 버그를 **발견한 후** 녹화를 시작하면 이미 버그 발생 순간은 지나가 있다. 재현해야 하고, 재현이 안 되면 영상이 쓸모없다.

Jam의 instant replay처럼 "항상 녹화 중, 필요할 때 직전 30초를 잡아냄"이 이상적이다. 단, Jam은 호스팅 뷰어 링크 방식이고 BugShot은 Jira/GitHub/Linear/Notion에 **MP4 파일을 직접 첨부**해야 하므로, 클라이언트에서 빠르게 인코딩까지 완료해야 한다.

## 목표

1. 사이드패널이 열려 있는 동안 현재 탭의 스크린을 상시 캡처한다.
2. 사용자가 "Instant Replay" 클릭 시 직전 30초를 H.264 MP4로 인코딩하여 **3초 이내에** 비디오 모드 드래프팅에 진입한다.
3. 기존 콘솔/네트워크 로그 첨부 흐름을 그대로 재사용한다.
4. 기존 수동 비디오 녹화 모드는 유지한다 (공존).

## 비목표 (Non-goals)

- rrweb DOM 녹화/재생: 검증 결과 픽셀 변환 비용이 비현실적. 제외.
- 서버사이드 인코딩: 프라이버시·인프라 이유로 제외.
- 오디오 캡처: 탭 오디오 불필요.
- 30초 초과 버퍼: 메모리 제약상 30초로 고정.
- rrweb 기반 인터랙티브 리플레이 뷰어: 스코프 외.

## 사용자 시나리오

### 시나리오 1: 최초 사용 (권한 미보유)

1. 사용자가 사이드패널을 연다.
2. "Instant Replay" 영역에 권한 요청 안내가 표시된다.
3. 사용자가 "Enable" 클릭 → `optional_host_permissions` 승인 팝업.
4. 승인 → 프레임 캡처 루프 자동 시작. 이후 패널을 열 때마다 자동.

### 시나리오 2: 일반 사용

1. 사이드패널이 열려 있고 프레임 캡처가 진행 중 (UI에 작은 indicator).
2. 사용자가 웹 페이지에서 버그를 발견한다.
3. "Instant Replay" 버튼 클릭.
4. 1-2초간 인코딩 진행 표시 → MP4 생성 완료.
5. 비디오 모드 드래프팅 화면 진입 (썸네일 + 타이틀 + 마크다운 에디터).
6. 콘솔/네트워크 로그가 자동 첨부 (기존 로직 동일).
7. 이후 흐름은 기존 비디오 모드와 동일 (프리뷰 → 플랫폼 제출).

### 시나리오 3: 캡처 버퍼 부족

1. 사이드패널을 연 직후 (3초 경과) "Instant Replay" 클릭.
2. 버퍼에 6장만 있으므로 3초짜리 짧은 MP4가 생성된다.
3. 정상 진행 — 최소 프레임 제한 없음.

### 시나리오 4: 탭 전환

1. 사용자가 다른 탭으로 전환.
2. `captureVisibleTab`이 다른 탭을 캡처하므로, 바운드 탭이 비활성이면 캡처 일시 중지.
3. 바운드 탭으로 돌아오면 캡처 재개.

### 시나리오 5: 권한 거부/미승인

1. 사용자가 권한을 거부하거나 승인하지 않음.
2. Instant Replay 비활성 상태. 기존 수동 비디오·스크린샷·요소 선택 모드는 정상 사용 가능.

## 성공 기준

- [ ] 사이드패널 열림 → 2fps 프레임 캡처 자동 시작 (권한 보유 시)
- [ ] "Instant Replay" 클릭 → 3초 이내 MP4 blob 생성 + 드래프팅 진입
- [ ] 생성된 MP4가 Jira 이슈 첨부 시 inline preview 재생 가능
- [ ] 메모리 사용량: 프레임 버퍼 ≤ 15MB
- [ ] 기존 수동 비디오 녹화 모드 정상 동작 (회귀 없음)
- [ ] 콘솔/네트워크 로그 첨부 기존 흐름과 동일하게 동작
