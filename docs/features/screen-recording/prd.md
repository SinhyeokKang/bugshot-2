# 화면 전체 녹화 (Screen Recording)

## 배경

현재 비디오 녹화는 `chrome.tabCapture`로 **현재 탭의 뷰포트만** 잡는다. 다음 한계가 있다:

- 브라우저 탭 바깥(다른 앱 창, OS 다이얼로그, 멀티 윈도우 플로우)을 보여줄 수 없다.
- 탭 경계를 넘나드는 재현(새 창 팝업, 외부 앱 연동)을 녹화로 전달할 방법이 없다.

실제 막힌 사례: 외부 결제창(다른 origin 팝업)·OS 권한 다이얼로그·새 창으로 뜨는 OAuth 플로우·데스크톱 앱 연동 등 "탭 밖에서 일어나는 일"을 QA·디자인 리뷰가 이슈에 담을 방법이 없어 글/스크린샷으로 우회해 왔다. 화면/창 단위 녹화가 이 공백을 메운다.

함께, 화면 녹화 추가로 녹화 진입로가 3종(탭 녹화·화면 녹화·30s 리플레이)이 되면서 idle 버튼 묶음을 재정렬한다 — 녹화 2종을 한 행(Row3)에 짝짓고 즉시성 도구(30s 리플레이)를 아래 단독 행으로 분리해, 새 버튼 추가가 기존 버튼 위계를 흐리지 않게 한다(재배치는 이 기능이 버튼을 1개 늘리는 데 따른 직접 후속 작업).

## 목표

- `getDisplayMedia` 기반 **화면 전체 녹화** 모드를 추가한다. 사용자가 브라우저 네이티브 picker에서 [전체 화면 / 특정 앱 창 / 탭] 중 대상을 고른다.
- 결과물은 기존 뷰포트 녹화와 **동일한 경로**(drafting Media 섹션, IndexedDB `video:{tabId}`, 제출 시 첨부)로 흐른다 — 사용자에게 "또 다른 녹화 영상"으로 일관되게 보인다.
- idle 화면 캡처 버튼을 **1×2×2×1** 레이아웃으로 재배치한다:
  - Row1(1): 요소 선택
  - Row2(2): 요소 캡처 · 영역 스크린샷
  - Row3(2): 탭 녹화 · 화면 녹화
  - Row4(1): 30s 리플레이 (단독·하단)
  - footer는 기존대로 가이드 · 프리폼 2열 유지.
- 버튼 라벨은 A안: 기존 `issue.mode.video`("화면 녹화/Record screen", 실제론 tabCapture라 오표기)를 **"탭 녹화/Record tab"**로 바로잡고, 신규 화면 녹화에 **"화면 녹화/Record screen"**을 부여한다.

## 비목표 (Non-goals)

- **오디오 녹음** 미포함. 영상만(`audio: false`) — 기존 뷰포트 녹화와 동일.
- **새 단축키** 미추가. Chrome `suggested_key` 4개 상한이 이미 꽉 차 있다(패널토글 + element/screenshot/video). 화면 전체 녹화는 버튼으로만 진입.
- **로그 첨부 정책 변경** 없음. 화면 전체 녹화도 현재 탭의 console/network/action 로그를 그대로 첨부한다(뷰포트 녹화와 동일). 다른 창을 녹화하면 영상↔로그 탭이 다를 수 있으나, 사용자가 drafting에서 로그 섹션을 끌 수 있어 수용한다.
- **새 captureMode 추가** 없음. `captureMode: "video"`를 공유한다 — 소스(tab vs screen)만 다르고 결과 처리·UI는 동일. 단, 녹화 중 화면의 "무엇을 찍는지" 안내를 위해 recorder source(`tab`/`screen`) 한 필드는 들고 라벨·아이콘만 분기한다(captureMode 분기 아님).
- **네이티브 picker 1스텝 추가 수용**. getDisplayMedia는 OS 권한 모델상 대상 선택+공유 클릭이 강제된다 — 기존 원클릭 캡처보다 마찰이 있으나 화면 캡처에 불가피하므로 수용한다. picker에서 "탭"을 고르면 기존 탭 녹화와 결과가 사실상 겹치지만(중복 경로) 막지 않는다(사용자 선택 자유).
- **60초 자동중지 상한 유지**. 탭 녹화와 동일한 60초를 상속한다. 화면 녹화가 더 길 수 있다는 점은 인지하나, 상한 확장은 이번 스코프 밖(별도 작업).
- **desktopCapture / 화면 영역 크롭 / 녹화 중 주석 / 화면 해상도 사용자 설정** 미포함. (화면 녹화 constraint는 1080p로 고정 — 용량 우선.)

## 사용자 시나리오

### 주 플로우: 화면 전체 녹화
1. 사용자가 디버그 탭 > 이슈 작성 idle 화면에서 **[화면 전체 녹화]** 버튼을 클릭한다.
2. 브라우저 네이티브 화면 공유 picker가 즉시 뜬다. 사용자가 [전체 화면 / 창 / 탭] 중 하나를 고르고 "공유"를 누른다.
3. 녹화가 시작되고 패널은 recording 화면(RecordingState — 경과 시간·중지 버튼)으로 전환된다. 화면 녹화일 때는 "화면 녹화 중", 탭 녹화일 때는 "탭 녹화 중"으로 source를 구분 표시한다(아이콘도 분기).
4. 사용자가 재현 동작을 수행한다(탭 밖 동작 포함).
5. 다음 중 하나로 녹화가 끝난다:
   - 패널의 **중지** 버튼 클릭
   - 브라우저 화면 공유 막대의 **"공유 중지"** 클릭 (track ended)
   - 60초 최대 길이 도달 (자동 중지)
6. drafting phase로 전환되어 Media 섹션에 녹화 영상이 표시되고, 현재 탭의 로그가 첨부된다.

### 엣지 케이스
- **picker에서 취소**: 사용자가 공유 picker를 닫으면(`getDisplayMedia`가 `NotAllowedError`로 reject) idle 상태 유지, **콘솔 경고 없이** 조용히 no-op(사용자가 스스로 취소했으므로 별도 피드백 불필요).
- **picker에서 "탭" 선택**: 영상↔로그가 같은 탭이라 불일치가 없는 정상 케이스. 결과는 탭 녹화와 사실상 동일하게 drafting으로.
- **공유 중지 버튼**: 브라우저가 제공하는 "공유 중지"를 누르면 video track의 `ended` 이벤트로 녹화가 정상 종료되어 drafting으로 간다.
- **getDisplayMedia 미지원/실제 실패**(취소 외 에러): 환경이 API를 막으면 `console.warn` 후 idle 유지(취소의 silent와 구분 — 실패만 경고). graceful.
- **녹화 중 패널 닫힘**: side panel이 닫히면 recorder state·스트림이 소멸해 진행 중 녹화가 손실된다(탭 녹화와 동일 한계). 화면 녹화는 다른 창으로 포커스를 옮기다 패널을 닫을 확률이 더 높으나, 자동 저장은 이번 스코프 밖 — 손실 수용.
- **사이드패널/민감정보 노출**: 전체 화면 선택 시 사이드패널 UI(토큰·이슈 내용 포함 가능)도 녹화될 수 있다. 무엇을 공유할지는 picker에서 사용자가 정하는 책임 영역이라 별도 차단/안내는 없음. privacy.md에 캡처 범위로 명시.

## 성공 기준

- idle 화면이 1×2×2×1 레이아웃으로 렌더되고, [화면 녹화] 버튼(`data-testid="mode-screen-record"`)이 탭 녹화와 같은 행(Row3)에, 30s 리플레이가 아래 단독 행(Row4)에 배치된다(DOM 구조로 판정 가능 — 같은 ButtonGroup 형제 vs 별도 컨테이너).
- [화면 녹화] 클릭 → getDisplayMedia picker → 대상 선택 → 녹화 → 중지 → drafting Media 섹션에 영상 표시 + 재생 가능.
- 녹화 중 화면이 source(탭/화면)에 따라 "탭 녹화 중"/"화면 녹화 중"으로 구분 표시된다.
- "공유 중지" / 패널 중지 / 60초 자동중지 세 경로 모두 drafting으로 정상 종료(세 경로 경합 시 중복 종료 없음).
- picker 취소가 idle을 깨지 않고 콘솔 에러를 남기지 않는다.
- 기존 뷰포트 녹화·30s 리플레이·요소/스크린샷 캡처가 회귀 없이 동작.
- `docs/privacy.md`에 화면 전체 녹화(탭 밖 데이터 캡처 가능)가 반영되고 시행일이 갱신된다.
