# 네트워크 로그 검색어 하이라이트

## 배경

로그 탭(console/network/action)은 이미 텍스트 검색을 제공하지만, 매칭 결과를 **필터링만** 한다 — 검색어에 걸린 행만 남기고, 정작 **어느 문구가 왜 매칭됐는지**는 표시하지 않는다. 특히 네트워크 로그는 `requestMatchesQuery`가 URL·요청/응답 헤더·요청/응답 바디까지 훑어 매칭하므로, 검색에 걸린 요청을 열어도 헤더·페이로드 어디에 검색어가 있는지 눈으로 다시 찾아야 한다. 상세는 헤더·JSON 트리·긴 바디로 정보 밀도가 높아 이 재탐색 비용이 다른 탭보다 크다.

검색어에 배경색 하이라이트를 입히면(브라우저 기본 검색·문서 검색 UI의 관례) 매칭 위치를 즉시 파악할 수 있다.

## 목표

- 네트워크 로그 **상세 패널**에서 현재 검색어와 일치하는 문구에 배경색(초록) 하이라이트를 표시한다.
- **하이라이트 대상은 실제 검색이 매칭하는 필드로 한정**한다(검색 표면 = 하이라이트 표면 일치). `requestMatchesQuery`가 훑는 URL·헤더(key/value)·바디만 대상:
  - **headers 탭**: General 섹션은 **URL만**(method·status·time·contentType은 검색 대상이 아니므로 제외) + 응답 헤더 + 요청 헤더(key/value)
  - **request 탭**: 요청 바디(payload) — plain text `<pre>` 및 JSON 트리
  - **response 탭**: 응답 바디 — plain text `<pre>` 및 JSON 트리
- **매칭은 정규식이 아니라 문자열(소문자 substring) 방식**이다 — 기존 네트워크 검색(`requestMatchesQuery`의 `.toLowerCase().includes()`)과 동일 규칙. 검색어의 정규식 특수문자는 리터럴로 취급된다.
- 검색어가 비어 있으면 하이라이트 없이 렌더되며 `<HighlightedText>`는 원문을 그대로 반환한다(하이라이트 계층 동작·성능 변화 0).

## 비목표 (Non-goals)

- **콘솔·액션 로그 하이라이트 제외.** 이 둘은 레벨(error/warn/info)·종류별 **행 배경색**을 이미 쓰고 있어, 하이라이트 배경이 겹치면 구분이 무너진다.
- **접힌(collapsed) 네트워크 로그 행의 URL 하이라이트 제외.** 상세를 펼쳤을 때만 하이라이트한다.
- **WebSocket `messages` 탭 하이라이트 제외.** 요청은 헤더·페이로드·리스폰스로 한정. (`FrameBody`가 동일 프리미티브를 쓰므로 후속 확장은 저비용이나 이번 스코프 아님.)
- **매치 네비게이션(다음/이전 이동·매치 개수 표시) 제외.** 검색이 이미 비매칭 요청을 목록에서 걸러내므로 단순 배경 하이라이트로 충분.
- 검색 로직·매칭 대상 필드 변경 없음(하이라이트는 순수 표시 계층).

## 사용자 시나리오

1. 사용자가 네트워크 로그 탭에서 검색창에 `token`을 입력한다.
2. 목록이 `token`을 URL·헤더·바디 어딘가에 포함한 요청만 남긴다(기존 동작).
3. 사용자가 그중 한 요청을 클릭해 상세를 연다.
4. **headers 탭**의 응답 헤더 `Authorization: Bearer …token…`에서 `token` 문구가 초록 배경으로 강조된다.
5. **response 탭**으로 이동하면 JSON 응답 트리에서 `"access_token"` 키·`"…token…"` 문자열 값의 매칭 부분이 초록 배경으로 강조된다. 매칭이 깊이 접힌 노드 안에 있으면 그 조상 아코디언이 최초 검색 시 자동으로 열려 매칭이 드러난다.
6. 검색창을 비우면(X 클릭) 하이라이트가 사라진다. (검색 중 자동으로 펼쳐졌던 아코디언은 사용자가 마지막에 둔 상태로 남는다 — 아래 collapse 존중 참조.)

### 엣지 케이스

- **URL로만 매칭된 요청**: 상세를 열면 General 섹션의 URL에만 하이라이트가 뜨고 헤더·바디에는 없다(정상 — 그 필드엔 검색어가 없음).
- **정규식 특수문자 검색어**(`api.v2(beta)` 등): 리터럴로 취급(문자열 매칭이라 이스케이프 불필요).
- **JSON 문자열 값이 300자 초과로 잘린 경우**(`StringRow`의 truncate): 현재 검색어가 그 값에 매칭되면 자동으로 전체가 펼쳐져 하이라이트가 보인다. 검색 중이 아니면 기존 truncate 동작 유지.
- **자동 펼침 후 collapse 존중**: 검색 시 매칭 조상 아코디언이 열리지만, 사용자가 그 노드를 접으면 그 상태를 존중한다(다시 강제로 열지 않음). 검색어를 바꾸면 새 매칭 기준으로 다시 최초 1회 펼친다.
- **마스킹된 값 제외**: 헤더 마스킹 값(`***`로 시작, italic muted 렌더)과 바디 JSON 마스킹 값(`"***"`) 모두 하이라이트 미적용.
- **본질적 한계(하이라이트 불가)**: ① 배열 원소가 100개(`ARRAY_CHUNK_SIZE`) 넘어 잘린 뒤쪽에 있는 매칭, ② JSON 구문·들여쓰기를 걸친 검색어(`"id":1`, `},{` 등)는 트리가 파싱된 구조를 키/값 별개로 렌더하므로 연속 문자열이 없어 하이라이트되지 않는다(헤더도 `"key: value"` 걸친 쿼리 동일). **목록엔 매칭으로 떠도 상세 트리엔 하이라이트가 0일 수 있다.** `<pre>` 경로(비-JSON 바디)는 raw 텍스트라 이 한계 없음.

## 성공 기준

- 네트워크 상세 3개 탭(headers/request/response)에서 검색어 매칭 문구가 초록 배경으로 표시된다(검색 대상 필드 한정).
- JSON 바디(트리 렌더)와 non-JSON 바디(`<pre>`) 양쪽에서 하이라이트가 동작한다.
- **빈 쿼리 시 `splitHighlight`가 하이라이트 span 없이 원문 단일 세그먼트를 반환한다**(단위 테스트로 검증 — 하이라이트 DOM이 삽입되지 않음).
- 콘솔·액션 로그, 접힌 네트워크 행, WS messages 탭에는 하이라이트가 나타나지 않는다.
- `splitHighlight` 순수 함수 단위 테스트가 통과한다(대소문자·정규식 특수문자 리터럴·다중 매칭·빈 쿼리·무매칭).
- `collectMatchExpandedPaths` 단위 테스트가 통과한다(depth 2+ 중첩 매칭 조상 path·빈 쿼리→빈 Set).
