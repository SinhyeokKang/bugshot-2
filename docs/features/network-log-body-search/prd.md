# 네트워크 로그 본문 검색

## 배경

네트워크 로그 탭의 검색창은 현재 요청 **URL만** 매칭한다(`NetworkLogContent.tsx:180-183`). 하지만 사용자가 버그를 추적할 때 찾고 싶은 건 종종 URL이 아니라 **요청 페이로드·응답 본문 안의 값**(에러 메시지, 특정 id, 필드명)이거나 **헤더 값**(content-type, set-cookie 등)이다.

데이터는 이미 다 캡처돼 있다(`NetworkRequest.requestBody` / `responseBody` / `requestHeaders` / `responseHeaders`). 검색 범위만 좁아서 활용을 못 하고 있다. 크롬 데브툴스의 Search 패널이 요청/응답 본문 전체를 grep하는 것과 같은 동작을 기본 필터창에서 제공한다.

## 목표

- 네트워크 로그 검색창 입력이 URL뿐 아니라 **request body, response body(둘 다 문자열일 때), request/response 헤더(키·값)** 까지 매칭한다.
- 매칭은 대소문자 무시, 부분 문자열(`includes`) 방식 — 기존 URL 검색과 동일한 UX.
- 본문이 비문자열(`{kind:"truncated"|"binary"|"stream"|"omitted"}`)이면 본문 항목은 건너뛰고 에러 없이 다른 필드로 진행한다.
- 큰 본문(최대 3MB×최대 5000건)에서도 타이핑이 끊기지 않도록 검색어 입력에 디바운스를 적용한다.

## 비목표 (Non-goals)

- 정규식·대소문자 토글·필터 키워드(`status:`, `method:` 등) 지원. (이번엔 단순 부분 문자열만)
- 매칭된 본문 라인 하이라이트·점프(데브툴스 Search 패널식 결과 표시). 매칭된 **요청 행**만 필터링한다.
- 콘솔 로그·액션 로그 탭 검색 변경. (이미 본문/타깃을 검색 중 — 범위 밖)
- 마스킹(`***`) 우회. token/password 등은 캡처 시점에 이미 마스킹돼 저장되므로 **마스킹된 값으로만** 검색된다(원문 검색 불가, 의도된 동작).
- 검색 결과 캐시·사전계산(precompute) 최적화. 디바운스로 충분하며, 부족하면 후속 과제.

## 사용자 시나리오

1. 사용자가 네트워크 로그 탭에서 검색창에 `"invalid_grant"`(응답 본문에 있는 에러 코드)를 입력한다.
2. ~200ms 후 필터가 적용돼, URL에는 없지만 응답 본문에 해당 문자열을 포함한 요청 행만 목록에 남는다.
3. 행을 클릭하면 기존대로 Response 탭에서 본문을 확인한다.

엣지 케이스:
- 검색어가 마스킹된 키(`password` 등)의 **원래 값**이면 매칭 안 됨(저장값이 `***`).
- 본문이 3MB 초과로 `truncated`거나 이미지 등 `binary`면 그 본문은 검색 대상에서 제외(다른 필드는 검색됨).
- 검색어가 헤더 키(`content-type`)거나 값(`application/json`)이면 매칭됨.

## 성공 기준

- 응답 본문에만 존재하는 문자열로 검색해 해당 요청이 필터링된다(단위 테스트로 검증).
- 비문자열 본문이 섞여 있어도 검색이 throw 없이 동작한다(단위 테스트).
- 검색어 입력 중 즉각적인 재필터링이 일어나지 않고 디바운스 후 1회 적용된다(수동/관찰).
- `pnpm test` 통과, i18n ko/en 대칭 유지.
