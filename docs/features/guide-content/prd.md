# BugShot 사용자 가이드 콘텐츠

## 배경

`guide/ko`·`guide/en`은 GitBook으로 공개되는 사용자 가이드의 in-repo 소스다(레포 → GitHub Sync 단방향). 현재 각 언어에 `README.md` 한 장만 있고 본문은 전부 `(작성 예정)/(coming soon)` 플레이스홀더다. 사이드패널에는 이미 항상 노출되는 "BugShot 가이드" 버튼 2곳(이슈 빈 화면 푸터, 설정 푸터)이 이 GitBook을 가리키므로, 사용자가 버튼을 눌러도 빈 문서로 떨어진다.

## 목표

- BugShot의 사용자 노출 기능 전체를 카테고리 기반 IA로 구조화하고, 각 페이지에 실제 동작 기반 콘텐츠를 채운다.
- `guide/ko`·`guide/en` 양쪽을 **동일 구조**로 동시 작성한다(키·파일 트리 대칭).
- 각 페이지에 스크린샷 자리를 더미 이미지로 채워, 사용자가 나중에 실제 캡처로 교체만 하면 되게 한다.
- bug-shot.com 랜딩의 톤앤매너(존댓말 `-습니다`/`-어요` 혼용, 캐주얼하면서 전문적, 공감형 단문)를 따른다.

## 비목표 (Non-goals)

- 실제 스크린샷 촬영·교체 — 더미 이미지만 배치하고 교체는 사용자가 직접 한다.
- GitBook 사이트 설정·배포·도메인 — 이미 연결되어 있다(`USER_GUIDE_URLS`).
- 사이드패널 코드 변경 — 가이드 진입 버튼은 이미 구현됨. 본 작업은 `guide/` 마크다운만 다룬다.
- 영상·GIF·인터랙티브 데모 — 정적 스크린샷 + 텍스트만.
- 가이드 페이지 내 실습(in-guide practice)용 picker 동작 설명 — 기능 사용법에 집중.

## 사용자 시나리오

1. 사용자가 사이드패널 푸터의 "BugShot 가이드"를 누른다 → 로케일(`ko`/`en`)에 맞는 GitBook이 새 탭으로 열린다.
2. 좌측 SUMMARY에서 큰 섹션(연동 설정 / 기본 설정 / 요소 / 스크린샷 / 녹화 / 로그)을 펼쳐 원하는 카테고리를 찾는다.
3. 섹션 개요 페이지에서 무엇을 다루는지 1~2문단으로 파악하고, 하위 문서 바로가기로 이동한다.
4. 각 하위 문서에서 화면 스크린샷 + 단계 설명을 보고 기능을 따라 한다.
5. "이슈 작성" 류 페이지는 각 캡처 모드별로 자기완결(반복 허용) — 사용자는 자기가 쓰는 모드 문서만 열어도 전체 흐름을 안다.

### 엣지/특수 케이스

- freeform(텍스트만 작성)은 별도 섹션 없이 `로그 > 실시간 로그` 문서 안에 흡수한다(콘솔/네트워크 탭에서 바로 "Start Draft").
- 30초 리플레이는 `녹화 > 30초 리플레이`에서 다루되, **설정에서 권한 허용이 선행되어야 함**을 명시하고 `기본 설정 > 이슈 설정`으로 교차 링크한다.
- 로그 뷰어(`logs.html`)는 이슈를 받아 처리하는 **개발자 관점**으로 작성한다(첨부된 리포트를 어떻게 읽는가).

## 성공 기준

- `guide/ko`·`guide/en` 양쪽에 23개 페이지가 동일 트리로 존재하고 `SUMMARY.md`가 모든 페이지를 링크한다.
- 모든 페이지에 플레이스홀더가 남지 않는다(`(작성 예정)` 0건).
- 스크린샷이 필요한 모든 자리에 더미 이미지 + 캡션이 들어가 있고, 이미지 경로가 GitBook에서 깨지지 않는다.
- ko/en 페이지가 1:1 대응(같은 파일 경로, 같은 섹션 구성).
- 기술적 사실(단축키, 필드명, 플랫폼 연결 방식, 본문 섹션 구성 등)이 현재 코드와 일치한다.
- **탐색 가능성**: 주요 사용 의도(요소 수정 후 등록 / 스크린샷 등록 / 녹화 등록 / 과거 초안 재열람 / 플랫폼 연결 / 30초 리플레이 켜기 / 로그만 보고)가 각각 SUMMARY 1뎁스에서 도달 가능하다.
