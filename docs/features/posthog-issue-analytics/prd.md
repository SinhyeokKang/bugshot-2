# PostHog 이슈 제출 집계

## 배경

BugShot에는 현재 분석·텔레메트리가 전혀 없다. 사용자가 확장을 통해 실제로 이슈를 얼마나 제출하는지, 어느 트래커·어느 캡처 모드가 많이 쓰이는지 알 수 있는 지표가 0이다. 출시 후 다운로드 수(스토어 통계)만 있고, 핵심 가치 행동인 "이슈 제출"의 발생량을 알 수 없어 제품 판단의 근거가 없다.

PostHog로 "이슈 제출" 이벤트를 익명 집계해 **제출 총건수**와 **플랫폼·캡처 모드 분포**를 본다.

## 목표

- 사용자가 BugShot으로 이슈를 제출(성공/실패)할 때마다 PostHog에 `issue_submitted` 이벤트 1건을 전송한다.
- 이벤트에 `platform`(jira/github/linear/notion/gitlab/asana), `capture_mode`(screenshot/video/element/freeform), `result`(success/failure) 속성을 붙인다.
- PostHog 대시보드에서 총 제출건수, 플랫폼별·캡처모드별 분포, 성공/실패 비율을 집계할 수 있다.
- 전송 실패가 이슈 제출 플로우에 어떤 영향도 주지 않는다(완전 fire-and-forget).
- 개인 식별 정보(제목·URL·본문·로그·계정 정보)를 일절 전송하지 않는다.

## 비목표 (Non-goals)

- 고유 사용자 수(MAU)·재방문·퍼널 측정. distinct_id를 **완전 익명**(이벤트마다 새 랜덤)으로 두므로 사용자 단위 분석은 하지 않는다.
- 옵트아웃 토글·동의 UI. 항상 수집한다(비식별 집계 카운트만이라).
- 제출 외 다른 행동(picker 사용, 스타일 편집, 녹화 등) 추적.
- posthog-js SDK 도입·autocapture·세션 리플레이.
- 빌드 모드(dev/store)별 프로젝트 분리. 단일 PostHog 프로젝트 키를 쓴다.

## 사용자 시나리오

이 기능은 백그라운드 동작으로, 사용자에게 노출되는 UI가 없다.

1. 사용자가 캡처 → 편집 → 제출 다이얼로그에서 플랫폼을 골라 "제출"을 누른다.
2. 제출이 **성공**하면: `issue_submitted` { platform, capture_mode, result: "success" }가 PostHog로 비동기 전송된다. 사용자는 평소처럼 성공 토스트/이슈 링크를 본다.
3. 제출이 **실패**하면: `issue_submitted` { platform, capture_mode, result: "failure" }가 전송된다. 사용자는 평소처럼 에러 토스트를 본다.
4. PostHog 키가 빌드에 주입되지 않은 경우(예: 로컬 dev, 키 미설정): 전송 자체를 건너뛴다(no-op). 제출 플로우는 동일하게 동작.

### 엣지 케이스

- PostHog 엔드포인트가 느리거나 4xx/5xx/네트워크 오류여도 제출 결과·UI에 영향 없음. 에러는 콘솔에만 경고로 남기고 삼킨다.
- 오프라인 상태에서 제출 실패 시에도 `result: "failure"` 전송을 시도하되 실패하면 조용히 무시.
- 같은 이슈를 재제출(중복)하면 그만큼 이벤트가 더 쌓인다. 중복 제거는 하지 않는다(제출 행위 자체를 카운트).

## 성공 기준

- 6개 플랫폼 각각으로 제출 성공 시 PostHog에 올바른 `platform`/`capture_mode`/`result=success` 이벤트가 1건씩 도착한다.
- 제출 실패 시 `result=failure` 이벤트가 도착한다.
- PostHog 키 미설정 빌드에서 전송이 일어나지 않고, 콘솔 에러 없이 제출이 정상 동작한다.
- 전송 경로 코드가 throw해도 제출 성공/실패 UI가 그대로 유지된다(단위 테스트로 격리 확인).
- 전송 payload에 제목·URL·본문 등 식별 정보가 포함되지 않는다(코드 리뷰 + 테스트로 확인).
