# 30초 리플레이 트리밍 (Replay Trim)

## 배경

30초 리플레이는 "녹화 버튼을 누르기 전 직전 30초"를 사후 캡처한다. 하지만 버그가 발생한 구간은 보통 그중 2~5초뿐이고, 나머지는 무관한 화면이다. 현재는 전체 30초를 그대로 인코딩·첨부하므로:

- 이슈에 붙는 영상이 불필요하게 길고 무겁다(전체 30초 ≈ 60프레임).
- 첨부 로그(console/network/action)도 30초 전 구간이라 잡음이 많다.
- 영상-로그 모두 버그 순간으로 좁혀줄 수단이 없다.

리플레이는 내부적으로 `{blob, timestamp}[]` 프레임 배열(`FrameBuffer`)을 들고 있어, 앞뒤 프레임을 잘라내고 다시 인코딩하는 비용이 작다. 이 구조 위에서 캡처 직후 한 번 트리밍 구간을 고르게 해 영상과 로그를 동시에 버그 구간으로 좁힌다.

## 목표

- 30초 리플레이 캡처 직후, drafting 진입과 동시에 **트리밍 다이얼로그**가 자동으로 떠 사용자가 보존할 구간(앞/뒤 잘라내기)을 in/out 핸들 드래그로 고른다.
- 확정(크롭) 시 잘린 프레임 구간만 **재인코딩**해 `videoBlob`을 교체하고, 첨부 로그를 새 영상 구간 경계로 **재trim**해 store·IndexedDB(`pending:${tabId}`)에 반영한다 — 즉시·파괴적.
- 트리밍 다이얼로그 UI(`<video>` + 듀얼 핸들 타임라인)는 영상 소스에 비종속적으로 설계해, 추후 일반 녹화(탭/화면) 트리밍이 같은 UI를 재사용할 수 있는 seam을 남긴다.

## 비목표 (Non-goals)

- **일반 녹화(탭/화면, MediaRecorder) 트리밍 구현**: 일반 녹화 결과는 불투명 Blob(mp4/webm)이라 프레임 접근이 불가하고, 트리밍하려면 WebCodecs `VideoDecoder` demux/decode → 재인코딩(또는 ffmpeg.wasm)이라는 별개 트랜스코드 파이프라인이 필요하다. 이번 스코프에서는 **다이얼로그 UI만 재사용 가능하게 설계**하고, 일반 녹화의 trim 적용 백엔드는 만들지 않는다.
- **확정 후 재편집**: 다이얼로그는 캡처 직후 단 한 번 뜬다. 확정 후 drafting에서 다시 트리밍하는 진입점은 만들지 않고, 원본 프레임·원본 로그는 **확정(적용) 즉시 폐기**한다(undo 불가). 단, **다이얼로그가 열려 있는 동안**에는 핸들 조정 undo/redo를 제공한다(확정 전 한정 — 다이얼로그를 닫으면 히스토리도 소멸).
- **프레임 단위보다 정밀한 트리밍**: 리플레이는 ~600ms 간격 60프레임이라 트리밍은 프레임 경계에 스냅된다(초 단위 핸들 → 가장 가까운 프레임 인덱스). 서브프레임 정밀도는 제공하지 않는다.
- **공간 크롭(영상 화면 영역 자르기)**: "크롭"은 시간축 앞뒤 잘라내기(trim)만 의미한다. 화면 영역 크롭은 비목표.
- 캡처 모드 추가·새 권한·새 외부 API 없음.

## 사용자 시나리오

### 주 플로우

1. 사용자가 IssueTab 진입 화면(`EmptyState`)에서 **30초 리플레이** 버튼을 누른다.
2. 기존대로 프레임 스냅샷 → 전체 인코딩 → 로그 trim → drafting 전환이 일어난다(로딩: "인코딩 중…").
3. drafting 화면이 뜨자마자 그 위로 **트리밍 다이얼로그**가 자동으로 뜬다.
4. 다이얼로그에는 방금 만든 전체 영상이 `<video>`로 재생되고, 하단 타임라인에 양쪽 in/out 핸들 2개가 있다(초기값: 전체 구간).
5. 사용자가 핸들을 드래그해 버그 구간만 남기고 앞뒤를 자른다. 핸들을 움직이면 영상이 해당 위치로 seek되어 경계 프레임을 확인할 수 있고, 트랙 하단에 선택 길이("3.4s / 30s")가 표시된다. 핸들 조정은 undo/redo로 되돌릴 수 있다.
6. **적용(크롭)**을 누르면: 재인코딩 동안 적용 버튼이 잠기고 스피너가 뜬다(`busy`). 선택 구간 프레임만 재인코딩 → `videoBlob` 교체, 로그를 새 구간 경계로 재trim, drafting의 미리보기·첨부가 좁혀진 결과로 갱신된다. 첨부 토글 상태(영상·로그 첨부 여부)는 그대로 유지된다. 다이얼로그가 닫히고 drafting 작성을 이어간다.
7. **전체 유지**(또는 닫기)를 누르면: 트리밍 없이 전체 30초 클립 그대로 drafting을 이어간다.

### 엣지 케이스

- **프레임 2개 미만(0개/1개)**: 트리밍 의미 없음 → 다이얼로그를 띄우지 않고 전체 클립으로 바로 drafting(기존 동작). (정상 캡처는 `isReady` 게이트로 항상 ≥10프레임이라 실사용에선 거의 도달 안 하는 방어 가드.)
- **전체 구간 선택 후 적용**: in=첫 프레임 && out=마지막 프레임이면 재인코딩 생략하고 전체 유지와 동일 처리(불필요한 재인코딩·로그 재trim 회피).
- **재인코딩 실패**: 토스트로 알리고 전체 클립을 유지한 채 다이얼로그를 닫는다(캡처 자체는 이미 성공해 drafting에 전체 영상이 있으므로 데이터 손실 없음).
- **다이얼로그 중 패널 닫힘/리로드**: 원본 프레임은 메모리에만 있어 사라진다 → 재오픈 시 trim 불가, drafting에는 전체 클립이 남는다(파괴 철학상 허용되는 손실).
- **최소 구간**: in/out이 같은 프레임으로 좁혀지지 않도록 **최소 2프레임** 보장(단일 프레임 인코딩의 degenerate clip 회피). 슬라이더는 `minStepsBetweenThumbs`로 두 핸들이 겹치지 않게 한다.

## 성공 기준

- 30초 리플레이 캡처 후 trim 다이얼로그가 자동으로 뜨고, in/out 핸들로 앞뒤를 잘라 적용하면 drafting 미리보기 영상 길이가 선택 구간으로 줄어든다.
- 적용 후 첨부되는 console/network/action 로그가 새 영상 구간 경계 밖 항목을 포함하지 않는다. **경계 정의**: `videoStartedAt/EndedAt`은 선택 구간의 raw 프레임 timestamp(첫 프레임 시작·마지막 프레임 표시 끝)이고, 로그 trim 경계는 기존 `replayLogBounds`(시작측 guard band 적용)를 그대로 따른다 — 영상-로그 sync 0점은 영상 시작(raw timestamp)에 맞춘다.
- "전체 유지"는 기존(트리밍 없는) 동작과 동일한 결과를 낸다.
- 트리밍 다이얼로그는 프레임·인코딩을 모르고 **초 구간만 props로 주고받는다**(소스 비종속 인터페이스) — 추후 일반 녹화가 다른 적용 콜백으로 같은 UI를 재사용할 수 있다.
- 초→프레임 인덱스 매핑·프레임 오프셋 계산 순수 함수에 단위 테스트가 있고 `pnpm test` 통과.
- 일반 녹화 캡처 흐름은 이번 변경의 영향을 받지 않는다(회귀 없음).
