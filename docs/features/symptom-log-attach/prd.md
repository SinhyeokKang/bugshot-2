# 발생 현상 로그 삽입 (Symptom Log Insert)

## 배경

버그를 제보한 팀 개발자가 문제되는 API response를 리포트에 담았는데, "이게 logs.html을 열어야만 보이냐"는 피드백을 남겼다. 확인 결과 맞다.

현재 이슈 본문(예: Jira ADF)에 들어가는 로그 정보는 요약 카운트 한 줄뿐이다 (`emitLogSummaryAdf`, `src/sidepanel/lib/buildIssueAdf.ts:263` — "네트워크: N건 (에러 X건)"). 실제 request/response body 등 payload는 첨부된 `logs.html`(gzip+base64)을 다운로드해 열어야만 보인다. 리포터가 "이 응답이 문제다"라고 트래커 본문에서 바로 보여줄 방법이 없다.

특히 status 코드로 에러를 자동 판별하는 접근은 범용적이지 못하다 — HTTP 200인데 body가 `{ "result": "FAILED" }`인 API를 놓친다. 따라서 어떤 로그를 노출할지는 사람(리포터)이 직접 골라야 한다.

### 기존 "로그 첨부"와의 구분 (중요)

DraftingPanel에는 이미 제목이 **"로그 첨부"**인 Section이 있다 — 이건 캡처된 로그 전체를 `logs.html` 첨부에 포함할지 토글하는 기능이다. 이번 신규 기능은 **개별 로그 1건을 에디터 본문에 코드블럭 텍스트로 삽입**하는 것으로, 목적·동작이 다르다. 사용자 혼동을 막기 위해 신규 기능의 노출 라벨은 "첨부"가 아니라 **"본문에 삽입"** 계열로 지정한다(i18n 키 `draft.insertLog`). 기존 `logs.html` 토글은 그대로 공존한다.

## 목표

- 발생 현상 및 기타 문단 섹션(기대 결과·비고) 에디터에서, 캡처된 네트워크/콘솔 로그 중 하나를 골라 **코드블럭 텍스트로 본문에 삽입**할 수 있다.
- 삽입된 로그는 이슈 트래커 본문에 코드블럭으로 그대로 렌더되어, logs.html을 열지 않아도 트래커에서 바로 보인다.
- 네트워크 로그는 endpoint·status·payload(request body)·response body를 담는다. payload는 "200인데 FAILED" 케이스 재현에 필수다.
- 8개 트래커 전부에서 코드블럭이 정상 렌더된다 (Notion 2000자 제약 포함해 깨지지 않는다).

## 비목표 (Non-goals)

- 로그를 "핀"·"활성화"하는 영속 상태를 만들지 않는다. 삽입은 1회성 에디터 커맨드이며, 삽입 후엔 그냥 마크다운 콘텐츠(중복·수정·삭제 자유). — 영속 상태를 만들면 마이그레이션·동기화 부담이 생기는데, 삽입은 그냥 텍스트라 그럴 필요가 없다.
- 여러 로그를 한 번에 다중 선택하지 않는다. 단일 선택 → 삽입. 여러 개 붙이려면 다이얼로그를 반복해서 연다. — 다중 선택은 상태 배열이 늘고 삽입=액션 1:1 단순성이 깨진다.
- 기존 `logs.html` 첨부 토글(`NetworkLogPreviewDialog`/`ConsoleLogPreviewDialog`의 attach/detach)을 대체·변경하지 않는다. 별개 기능으로 공존한다.
- **action 로그는 삽입 대상이 아니다** (네트워크·콘솔만). — action 로그는 자연어 요약이라 코드블럭 인용의 가치가 낮고, "문제된 응답"이라는 이 기능의 동기와 무관하다.
- **WebSocket 프레임을 직렬화하지 않는다.** WS 항목은 목록엔 보이되 바디 없이 헤더 라인만 삽입. — WS는 request/response body 개념이 없고(프레임 스트림) 이 기능의 API-응답 초점 밖이라, 프레임 직렬화는 스코프 크리프다.
- **삽입 시 민감정보 경고 UX를 만들지 않는다.** 노출 판단은 사용자 책임으로 두되, 리스크를 명시한다(아래 privacy 항목).

## Privacy 영향 (코어밸류)

이 기능은 캡처된 response 원문을 **트래커 본문 평문**으로 승격시킨다. 네트워크 body는 캡처 시 `token/password/secret` 키만 마스킹되고, 그 외 payload(PII·전체 응답)는 원문이다. 지금까지는 logs.html(다운로드해야 열림)에 묻혀 있던 원문이 이슈 열람권자 전원에게 노출되고 AI 드래프팅 재료로도 흘러갈 수 있다.

- 삽입되는 로그는 **캡처 시 마스킹 규칙만 적용**되며, 그 외 본문은 원문 그대로 노출된다. 사용자가 상세 패널로 내용을 확인한 뒤 스스로 삽입한다.
- 캡처 데이터는 여전히 BugShot 서버를 거치지 않는다(클라이언트 내 텍스트 조작) — 데이터 직행 구조는 유지.
- **CLAUDE.md 문서 신선도 규칙**상 "기존 권한을 새 목적으로 쓰거나 새 노출 동작 추가 시 privacy 갱신"이 트리거된다. `docs/privacy.{ko,en}.md`를 대조·갱신한다(tasks.md 참조).

## 사용자 시나리오

### 주 플로우 — 네트워크 응답 삽입
1. 리포터가 drafting 화면에서 발생 현상 섹션을 작성 중이다.
2. 섹션 헤더 우측 버튼그룹의 **로그 삽입 버튼**을 누른다.
3. 다이얼로그가 열린다. 상단 탭으로 네트워크/콘솔 전환, 하단은 기존 로그 뷰(필터·검색·origin 필터·좌측 리스트 / 우측 상세).
4. 문제된 요청을 클릭하면 하이라이트되고 우측에 상세(headers/request/response)가 뜬다. 응답을 눈으로 확인한다.
5. **삽입** 버튼을 누른다.
6. 다이얼로그가 닫히고, 발생 현상 에디터 커서 위치에 아래 형태의 코드블럭이 삽입된다:
   ```
   POST /api/orders/123 → 200 OK
   --- payload ---
   { 정렬된 request body }
   --- response ---
   { "result": "FAILED", ... }
   ```
7. 제출하면 트래커 본문에 코드블럭으로 그대로 나타난다.

### 기타 문단 섹션
- 발생 현상 외에 **기대 결과·비고** 섹션에서도 같은 버튼·다이얼로그로 삽입 가능하다(문단 섹션 공통).

### 콘솔 로그 삽입
- 콘솔 탭에서 에러/경고 항목 선택 → 삽입 → `[error] 메시지` + (error면 stack) 형태 코드블럭 삽입.

### 엣지 케이스
- **로그 없음**: 캡처된 네트워크·콘솔 로그가 둘 다 없으면 버튼은 비활성(disabled).
- **body 없음(GET 등)**: payload/response 섹션 중 없는 것은 생략, 헤더 라인만.
- **body가 JSON 아님**: 정렬 실패 시 raw 텍스트 그대로.
- **body descriptor**(truncated/binary/stream/omitted): 원문이 없으므로 `[truncated 5MB/1MB]` 같은 라벨 한 줄로 대체.
- **16K chars 초과 body**: 16384자에서 자르고 `…(truncated)` 표시(문자 기준). 콘솔 args도 동일 캡.
- **pending·status 0 요청**: 헤더 라인이 `→ 0`·pending으로 깨지지 않게 status 표기 방어.
- **WebSocket 항목**: 목록엔 보이되, 삽입 시 `WS <url> → 101` 같은 헤더 라인만 (payload/response 섹션 없음).
- **Notion 제출**: 2000자를 넘는 코드블럭도 청킹으로 정상 제출된다.

## 성공 기준

- [ ] 발생 현상·기대 결과·비고 섹션 각각에서 네트워크 로그 1건을 골라 삽입하면 지정 포맷 코드블럭이 에디터에 들어간다.
- [ ] 콘솔 로그 1건을 골라 삽입하면 `[level] args`(+error면 stack) 코드블럭이 들어간다.
- [ ] 삽입된 리포트를 Jira·GitHub·Linear·Notion·GitLab·Asana·ClickUp·Slack에 제출하면 코드블럭이 본문에 정상 렌더된다 (특히 Notion에서 2000자 초과 시에도 400 없이 성공).
- [ ] 삽입된 코드블럭은 일반 텍스트라 이후 자유롭게 편집·삭제 가능하다.
- [ ] 신규 "로그 삽입" 버튼·다이얼로그 라벨이 기존 "로그 첨부"(logs.html)와 명확히 구분된다.
- [ ] 로그가 둘 다 없으면 버튼이 비활성이다.
- [ ] 순수 직렬화 함수 단위 테스트 통과: JSON 정렬 / non-JSON raw / body 없음 / descriptor 4종 / 16K 초과 truncate / pending·status0 / WS 헤더만 / 콘솔 error+stack / 콘솔 error-no-stack / 콘솔 비-error.
- [ ] Notion `richText` 청킹 단위 테스트 통과(빈 문자열/2000 이하/2001/16K 경계).
- [ ] `docs/privacy.{ko,en}.md`가 이 노출 동작을 반영해 갱신된다.
