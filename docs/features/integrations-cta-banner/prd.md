# 연동 CTA 배너 — 온보딩 퍼널 개선

## 배경

현재 `App.tsx`는 사이드패널이 hydration을 마쳤을 때 연결된 플랫폼이 0개면 `integrations` 탭으로 자동 전환한다.

```tsx
// src/sidepanel/App.tsx:102-106
useEffect(() => {
  if (settingsHydrated && connectedPlatforms(accounts).length === 0) {
    setTab("integrations");
  }
}, [settingsHydrated, accounts]);
```

이 동작은 가치 경험 전에 비용을 먼저 청구한다. 설치 직후 사이드패널을 처음 여는 사용자가 보는 첫 화면은 캡처 버튼이 아니라 OAuth 로그인 요구다. 그 사람은 아직 Bugshot이 스크린샷을 어떻게 찍는지, 콘솔·네트워크 로그를 얼마나 잘 수집하는지 아무것도 보지 못했다. OAuth는 심리적으로 싼 행동이 아니다 — 워크스페이스를 고르고, 권한 화면을 읽고, 회사 계정이면 관리자 승인까지 기다려야 한다.

게다가 이 전환은 1회성이 아니다. deps에 `accounts`가 있어 사이드패널을 닫았다 열 때마다(재마운트) 연동 0개 사용자는 계속 `integrations` 탭으로 끌려간다.

한편 연동 없이도 제품의 핵심 가치 경험은 이미 온전히 동작한다. 캡처 → 초안 작성 → 미리보기 → 클립보드 복사까지 아무 게이팅이 없고, 오직 "플랫폼에 이슈 제출"만 비활성화된다. 즉 앞단에서 연동을 요구할 기능적 이유가 없다.

그리고 전환 유도 장치는 이미 뒤쪽에 깔려 있다. `PreviewPanel`은 연동 0개일 때 안내 Alert를 띄우고, `IssueCreateModal`은 제출 버튼을 disabled 처리한다. 앞단의 탭 리다이렉트는 그 위에 얹힌 중복 요구다.

## 왜 지금

Product Hunt 런칭을 준비 중이다 (`7f3c81c` — "Product Hunt launch kit").

이 문제는 설치 이래 상시 존재했지만, 런칭은 **첫인상이 한 번에 몰리는 순간**이다. 유입된 사람이 사이드패널을 처음 열었을 때 캡처 버튼 대신 OAuth 로그인 요구를 보면, 제품이 무엇을 하는지 보기도 전에 이탈한다. 평소라면 점진적으로 개선할 문제지만 런칭 트래픽은 재방문을 기대하기 어려워, 그 순간의 첫 화면이 곧 활성화율이다.

따라서 이 변경은 런칭 전에 들어가야 한다.

## 목표

1. 사이드패널 최초 진입 시 연동 여부와 무관하게 `debug`(캡처) 탭에 착지한다.
2. 연동이 없어 이슈를 제출할 수 없는 화면에서, 연동 탭으로 한 번에 이동하는 CTA 배너를 노출한다.
3. 배너는 무시 가능하다 — 어떤 탭도 disabled 되지 않고, 어떤 플로우도 차단되지 않는다. 다만 **dismiss(닫으면 다시 안 뜸)는 제공하지 않는다** — 아래 "영구 노출 결정" 참조.
4. 기존 안내 Alert를 배너로 대체해, 같은 상태를 알리는 **안내 UI**가 두 벌 공존하지 않게 한다.

> 목표 4의 "안내 UI"는 `IssueCreateModal`의 disabled 제출 버튼을 포함하지 않는다. 배너는 *안내·유도*이고 disabled 버튼은 *제출 차단*이다. 목적이 다르므로 previewing 화면에서 둘이 함께 보이는 것은 모순이 아니다.

## 영구 노출 결정

배너는 연동이 생길 때까지 캡처·미리보기 화면에 계속 노출된다. 클립보드 복사만 쓰는 사용자에게는 매 세션 보인다는 뜻이다.

dismiss 어피던스를 주지 않는 이유는, dismiss가 `chrome.storage` 영속 상태와 마이그레이션을 끌어들이기 때문이다. 이는 design.md 대안 1(온보딩 플래그)을 기각한 사유와 정확히 같다. 그리고 연동은 이 제품의 부가 기능이 아니라 핵심 가치 제안이므로, 연동이 없는 상태를 계속 알리는 것은 나그가 아니라 정확한 상태 표시다. 자동 탭 전환과 달리 플로우를 차단하지 않고 한 줄 배너에 그친다.

**의도된 트레이드오프다.** 클립보드 전용 사용자의 반복 노출 피로를 감수한다.

## 비목표 (Non-goals)

- `IssueCreateModal`의 disabled 제출 버튼을 CTA로 바꾸지 않는다. 현행 유지(툴팁 포함).
- 다단계 온보딩 위저드·투어·툴팁 코치마크를 도입하지 않는다.
- 활성화율 A/B 실험 인프라를 만들지 않는다. 현재 트래픽(누적 100 다운로드)에서는 통계적 판정이 불가능하다.
- **배너 클릭 계측(PostHog `cta_click` 등)을 추가하지 않는다.** A/B 인프라와 별개로 절대 퍼널(배너 클릭 → `platform_connect`) 관찰은 가능하지만, 이번 스코프에서는 넣지 않기로 **의도적으로 결정**했다. 따라서 이 변경의 핵심 가설("자동 전환이 활성화율을 낮춘다")은 직접 검증되지 않으며, 성공 기준은 전부 기능 판정이다. 나중에 배너가 죽은 요소인지 확인하고 싶어지면 그때 `captureEvent` 한 줄을 얹는다.
- 배너에 dismiss(닫기) 어피던스를 주지 않는다. 위 "영구 노출 결정" 참조.
- 연동 0개 상태의 다른 화면(로그 탭, 설정 탭)에 배너를 확산하지 않는다.
- `drafting` 상태(`DraftingPanel`)에는 배너를 넣지 않는다 — 그 자리는 purple AI 배너가 이미 점유한다.

## 사용자 시나리오

### S1. 신규 설치 사용자 (핵심 경로)

1. 확장을 설치하고 사이드패널을 연다.
2. **`debug`(캡처) 탭에 착지한다.** 캡처 진입 화면(`EmptyState`)이 보인다.
3. 화면 하단 `PageFooter` 바로 위에 amber 배너가 있다: "플랫폼을 추가해 이슈를 등록하세요" + "플랫폼 추가".
4. 사용자는 배너를 무시하고 요소 선택·스크린샷·영상 중 원하는 캡처를 바로 시도할 수 있다.
5. 초안을 작성하고 미리보기(`previewing`)에 도달한다.
6. 미리보기 화면에도 같은 amber 배너가 있다. 이 시점이 전환 의도가 가장 높다 — 사용자에겐 이미 만들어진 리포트가 있다.
7. 배너를 클릭하면 `integrations` 탭으로 이동한다.

### S2. 연동을 미루는 사용자

1. 배너를 무시하고 리포트를 완성한다.
2. 미리보기에서 마크다운/HTML 클립보드 복사를 사용한다 (연동 여부와 무관하게 항상 제공).
3. 제출 버튼은 disabled이고 툴팁으로 "연결된 플랫폼이 없습니다"가 뜬다 (현행 유지).

### S3. 저장된 초안을 다시 여는 사용자

1. 이슈 목록에서 저장된 초안을 열어 `DraftDetailDialog`에 진입한다.
2. 제출 가능한 플랫폼이 없으면 `DialogFooter` 위에 같은 amber 배너가 뜬다.
3. 배너를 클릭하면 다이얼로그가 닫힌 뒤 `integrations` 탭으로 이동한다.

> 탭 전환이 다이얼로그를 자동으로 닫아주지는 않는다. 배너의 `onClick`이 `onOpenChange(false)`를 먼저 명시 호출한 뒤 `navTo`를 부른다 (design.md 참조).

### S4. 엣지 — Slack만 연동한 사용자 + Slack-preserved 이슈

`DraftDetailDialog`의 배너 노출 조건은 `available.length === 0`이고, `available`은 `submittablePlatforms(issue, accounts)`다. Slack-preserved 이슈에서는 `promotableTargets`(Slack 제외 연결 플랫폼)로 계산되므로, **Slack만 연동한 사용자는 연동이 0개가 아님에도 `available.length === 0`이 된다.**

현재 이 경우 Alert 제목 "연결된 플랫폼이 없습니다"가 뜨는데 이는 사실과 다르다(기존 버그). 배너는 제목 없이 본문("플랫폼을 추가해 이슈를 등록하세요")만 쓰므로 이 사용자에게도 참인 문장이 되고, 클릭 시 연동 탭으로 보내는 것도 올바른 조치다(트래커를 붙여야 승격 가능). 조건은 `available.length === 0` 그대로 유지한다.

### S5. 엣지 — 배너 노출 중 연동이 생기는 경우

다른 창에서 OAuth를 마치고 돌아오면 `accounts`가 갱신되어 배너가 즉시 사라진다. `useSettingsStore` 구독으로 자동 처리된다.

## 성공 기준

- 연동 0개 상태로 사이드패널을 열면 `tab-debug`가 active다. `tab-integrations`로 자동 전환되지 않는다.
- 사이드패널을 닫았다 다시 열어도 `tab-debug`에 착지한다.
- 캡처 진입 화면에 배너가 노출된다.
- **미리보기(`previewing`) 화면에 배너가 노출된다** — 전환 의도가 가장 높은 지점이므로 e2e 자동 검증 필수 항목이다(수동 테스트로 대체 불가).
- 배너 클릭 시 `tab-integrations`가 active가 된다.
- 연동이 1개 이상이면 배너가 노출되지 않는다.
- 4개 탭 어느 것도 disabled되지 않는다.
- 캡처 → 초안 → 미리보기 → 클립보드 복사 플로우가 연동 0개 상태에서 그대로 동작한다.
- `PreviewPanel`·`DraftDetailDialog`의 기존 Alert가 사라지고 배너로 일원화된다.
- 배너가 키보드 Tab으로 도달 가능하고, 도달 시 focus 링이 보인다.

> 위 기준은 전부 "제대로 만들었나"를 판정한다. "활성화율이 실제로 올랐나"는 이번 스코프에서 측정하지 않는다(비목표 참조).
