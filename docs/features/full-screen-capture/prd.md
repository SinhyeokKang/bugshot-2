# 전체 화면 캡처 (Full Screen Capture)

## 배경

스크린샷 모드는 현재 **영역 드래그 하나뿐**이다. 사이드패널에서 [스크린샷]을 누르면 `capturing` phase로 들어가고, 페이지에 크로스헤어 blocker가 깔리며, 사용자는 반드시 드래그해서 사각형을 그려야 한다(`area-select.ts`가 10px 미만 드래그는 무시).

하지만 실제 버그 리포트에서는 "보이는 화면 전체"나 "페이지 전체(스크롤 포함)"를 그대로 담고 싶은 경우가 잦다. 지금은 뷰포트 끝에서 끝까지 정확히 드래그해야 하고(가장자리 픽셀을 놓치기 쉽다), 스크롤 아래 내용은 아예 담을 방법이 없다.

진입(idle) 화면에 캡처 모드 버튼을 늘리는 건 부담스럽다 — 이미 element / screenshot / element-shot / 녹화(탭·화면) / replay 버튼과 freeform 진입이 경합한다. 대신 **이미 스크린샷 모드에 들어온 뒤(`capturing` phase) 캡처 방식을 고르게** 한다. 녹화 중 사이드패널 하단에 그리기 툴바가 뜨는 것과 **같은 자리(하단 footer 툴바)**다.

## 목표

스크린샷 모드의 `capturing` 단계에서 **3축 캡처**를 지원한다.

| 방식 | 동작 | 결과 |
|---|---|---|
| **영역 선택** (기존, 기본값) | 페이지를 드래그해 사각형 지정 | 드래그한 영역 |
| **뷰포트 캡처** (신규) | 버튼 1회 클릭 | 현재 보이는 화면 전체 |
| **스크롤 캡처** (신규) | 버튼 1회 클릭 → 자동 스크롤·연속 캡처·합성 | 페이지 전체(세로 스티칭) |

- 캡처 방식 선택 UI는 `capturing` 단계 **사이드패널 하단 footer 툴바**의 아이콘 버튼 3개(ButtonGroup)다. [영역 선택]이 활성 상태로 시작한다.
- idle(진입) 화면의 버튼 개수는 **그대로 유지**한다.
- 어떤 방식이든 캡처 결과물에 picker 오버레이(dim 4분면·선택 사각형·크기 라벨)가 **찍히지 않는다**.
- 브라우저 줌이 100%가 아니어도 캡처 가장자리에 빈 픽셀이 생기지 않는다(이번 3축의 크롭 경로 한정 — 아래 비목표 참조).
- 스크롤 캡처는 수 초가 걸리므로 **진행 상태(로딩)를 사이드패널에 표시**하고, **진행 중 [취소]로 중단**할 수 있다(중단 시 페이지 원상 복구 + idle 복귀).

## 비목표 (Non-goals)

- **가로 스크롤 스티칭** — 세로만. 가로로 넘치는 페이지는 뷰포트 폭까지만 담는다.
- **내부 스크롤 컨테이너(`overflow: auto` div) 캡처** — 스크롤 캡처는 문서 스크롤(`document.scrollingElement`)만 대상으로 한다.
- **element-shot 경로의 줌 보정** — 크롭 클램프 가드는 area/inline 캡처가 쓰는 `usePickerMessages.ts`의 로컬 `cropImage`에만 적용한다. element-shot은 별도 `capture.ts:cropImage`를 쓰며 이번 스코프에서 건드리지 않는다.
- **인라인 영역 캡처(drafting 중 본문 이미지 삽입, `DraftingPanel.tsx:349-368`)에 같은 툴바 추가** — 같은 `picker.startAreaSelect` 경로를 쓰지만 UI는 별도 컴포넌트다. 1차 제외.
- **스크롤 캡처 중 in-page Esc 취소** — 취소는 사이드패널 [취소] 버튼으로만. Esc 취소는 영역 선택 모드 한정(기존 동작).
- element 모드 / element-shot / video / freeform 흐름 변경.
- 페이지 오버레이 안에 in-page 버튼·힌트 텍스트 추가.
- 캡처 단축키 부활.

## 사용자 시나리오

### 주 플로우
1. 사이드패널 진입 화면에서 [스크린샷] 클릭 → `phase: capturing`, 페이지에 크로스헤어 blocker.
2. 사이드패널: "캡처 영역을 선택하세요" 안내 + [취소] 버튼(기존) + **하단 툴바에 캡처 방식 버튼 3개**. **[영역 선택]이 활성 상태**로 시작한다(현재 페이지에 오버레이가 깔린 그 모드).
3. 분기
   - 그대로 페이지를 드래그 → 기존과 동일하게 영역 캡처 → `drafting`.
   - **[뷰포트 캡처]** 클릭 → 오버레이가 걷히고 즉시 뷰포트 전체 캡처 → `drafting`.
   - **[스크롤 캡처]** 클릭 → 툴바가 로딩 상태(스피너 + `n/N` 진행)로 바뀌고, 페이지가 자동 스크롤되며 타일을 순차 캡처 → 합성 → `drafting`.
4. `drafting` — 캡처 이미지가 어노테이션 가능한 상태로 초안 화면에 올라온다.

### 스크롤 캡처 세부
- 시작 시 선택 오버레이(dim 4분면·사각형·라벨)는 걷히지만 **투명 blocker는 남아 페이지 클릭을 차단**한다(휠 스크롤은 가능). blocker는 투명이라 캡처에 찍히지 않는다.
- 고정 요소(`position: fixed` / `sticky` — 헤더·플로팅 채팅 버튼 등)는 **첫 타일에만 남기고 이후 타일에서는 숨긴다**. 캡처가 끝나면 원상 복구한다.
- 캡처가 끝나면 **원래 스크롤 위치로 복원**하고 blocker를 걷는다.
- **진행 중 [취소] 클릭 → 즉시 중단**: 페이지 스크롤·고정 요소·blocker를 원복하고 idle로 복귀한다.
- 타일 수 **상한(20타일)**. 초과하면 거기까지만 담고 사이드패널에 "페이지가 길어 일부만 캡처했습니다" 안내를 띄운다.

### 엣지 케이스
- **드래그 완료 직후 버튼 클릭(레이스)**: 이미 `picker.areaSelected`가 발화해 drafting으로 넘어가면 `CapturingState`가 언마운트되어 툴바가 사라진다. 메시지가 늦게 도착해도 content script의 area handle이 이미 null이라 no-op.
- **캡처 실패(activeTab 만료 / captureVisibleTab rate-limit)**: 기존 드래그 경로와 동일하게 권한 만료 안내를 띄우거나 `reset()`으로 idle 복귀. 스크롤 캡처 중 실패 시 **페이지 스크롤·고정 요소·blocker를 반드시 원복**한 뒤 idle로 돌아간다.
- **브라우저 줌 ≠ 100%**: 뷰포트 CSS px × DPR로 계산한 크롭 rect가 캡처 이미지 경계를 넘을 수 있다 → 크롭 rect를 이미지 경계로 클램프해 빈 픽셀을 막는다.
- **스크롤 캡처 중 사용자 휠 스크롤**: blocker가 클릭은 막지만 휠 스크롤은 통과한다. 각 타일 캡처 전에 목표 y로 다시 스크롤하지만, 스크롤 명령 응답과 실제 캡처 사이(캡처 큐 대기 ≥500ms)에 휠이 개입하면 **해당 타일이 어긋난 위치에서 찍힐 수 있다 — 허용 리스크**(재캡처로 해소, 결과물이 손상될 뿐 상태 오염은 없음).
- **스크롤 캡처 중 탭 전환**: `captureVisibleTab`은 창의 현재 보이는 탭을 찍으므로 다른 탭 화면이 타일에 섞일 수 있다 → 각 타일 캡처 전 `tab.active`를 확인해 비활성이면 **중단(abort)**하고 원복한다.
- **스크롤 캡처 중 페이지 네비게이션 / 사이드패널 닫힘**: 사이드패널 오케스트레이터가 죽으면 `finally` 복원이 실행되지 않는다 → content script가 **자가 복원**한다(picker port 단절·`handleClear` 시 열린 스크롤 세션을 스스로 원복). content script 자체가 소멸하는 네비게이션은 새 문서 로드로 자연 복원된다.
- **capturing reset 경합**: 세션 삭제·탭 URL 변경·picker port 단절이 capturing phase를 reset하는 기존 분기 3곳이 스크롤 캡처 도중 발화할 수 있다 → 캡처 완료 시 phase가 여전히 `capturing`인지 재확인하고 아니면 결과를 폐기한다(유령 drafting 진입 차단).
- **Esc**: 영역 선택 모드에서는 기존과 동일하게 취소. 스크롤 캡처 진행 중에는 무반응(취소는 사이드패널 [취소] 버튼).
- **iframe 내부 고정 요소**: 스크롤 캡처 메시지는 top frame(`frameId: 0`) 한정이라 1-depth iframe 내부의 fixed/sticky 요소는 숨겨지지 않아 타일마다 반복될 수 있다 — 허용.
- **미지원 페이지 / content script 미주입**: `startAreaCapture`가 이미 `ensureSupportedTab`으로 걸러 `capturing`에 진입하지 않는다. 툴바가 노출될 일이 없다.

## 성공 기준

- [ ] 스크린샷 모드 capturing 단계 사이드패널 하단에 아이콘 버튼 3개([영역 선택][뷰포트 캡처][스크롤 캡처])가 보이고, 영역 선택이 활성 상태다.
- [ ] [뷰포트 캡처] 클릭 → 드래그 없이 drafting 진입, 캡처 이미지가 뷰포트 전체다.
- [ ] [스크롤 캡처] 클릭 → 로딩(진행 표시) 후 drafting 진입, 캡처 이미지가 페이지 전체(세로)다.
- [ ] 스크롤 캡처 진행 중 [취소] 클릭 → 중단되고 페이지(스크롤·고정 요소·blocker)가 원복되며 idle로 복귀한다.
- [ ] 스크롤 캡처 결과에 `position: fixed` 고정 요소가 **1번만** 나온다(타일마다 반복되지 않음). sticky 숨김이 본문을 지우는 사이트가 발견돼 "fixed만 숨김" fallback을 채택하면 이 기준은 fixed 한정으로 조정된다(sticky는 반복 허용).
- [ ] 스크롤 캡처 정상 완료 후 페이지 스크롤 위치·고정 요소 스타일이 원래대로 복구된다.
- [ ] 어떤 방식이든 캡처 이미지에 dim·선택 사각형·크기 라벨이 없다.
- [ ] 줌 80% / 100% / 150%에서 캡처 가장자리에 빈(투명) 픽셀이 없다.
- [ ] idle 화면 버튼 구성이 변하지 않았다.
- [ ] 기존 드래그 영역 캡처 e2e(`e2e/capture.spec.ts`)가 계속 통과한다.
