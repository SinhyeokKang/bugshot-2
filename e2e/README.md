# e2e 스위트 (Playwright)

Chrome 확장을 실제 브라우저에서 구동해 사용자 플로우를 검증하는 e2e 스위트. **무엇을 커버하는지·무엇이 빠졌는지·어떤 함정이 있는지의 단일 출처**다. 시나리오가 늘면 이 문서를 함께 갱신해 "어디에 뭐가 있나"를 재조사하지 않게 한다.

- **작성 절차·금지·실행-수정 루프는 `/e2e-write` 스킬**(`.claude/commands/e2e-write.md`)이 단일 출처 — 여기서 중복하지 않는다. 이 문서는 **커버리지 맵 · 수동 잔여 · 함정 · 헬퍼 참조**만 담는다.
- 빌드/실행: `pnpm build:e2e` → `pnpm test:e2e` (단일 spec: `pnpm test:e2e -- <이름 일부>`). dist-e2e는 **테스트 전용**(`<all_urls>` 포함, 수동 로드·스토어 업로드 금지).
- **두 project로 분리**(`playwright.config.ts`): `sidepanel`(확장 구동, `e2e/*.spec.ts`)과 `logview`(확장 없이 `dist-log-viewer/index.html`을 합성 데이터로 직접 여는 standalone, `e2e/logview/*.spec.ts`). `logview`는 `dependencies: ["sidepanel"]`이라 사이드패널 green 후에만 돈다. logview만 빠르게: `pnpm test:e2e --project=logview --no-deps`(dist-log-viewer는 `build:log-viewer`/`build:e2e`가 생성).
- **창 깜빡임**: 확장 SW가 headless에선 안 깨어나 headed로만 돈다. 대신 브라우저 창을 화면 밖으로 보내 기본적으로 안 보인다. 디버깅으로 창을 직접 보려면 `E2E_SHOW=1 pnpm test:e2e`.

## 수동 잔여 (자동화 못 한 것 — 이유 포함)

스크립트로 판정 불가하거나 e2e 환경 제약으로 빠진 항목. 새로 자동화하면 위 맵으로 옮긴다.

## 함정 (gotchas — 실전에서 밟은 것 누적)

새 spec 쓰기 전에 훑어 같은 함정을 반복하지 않는다.

- **spec 간 탭 누수**: worker fixture(persistent context)를 모든 spec이 공유한다. `beforeAll`로 연 탭/패널은 **반드시 `afterAll`로 닫는다**. 안 닫으면 후행 spec의 `fixtureTabId()`가 잔여 탭을 잡아 **실행 순서에 따라 실패**한다(파일명 알파벳 순 실행 — rename으로 순서가 바뀌면 드러난다).
- **`fixtureTabId` 모호성**: fixture 탭이 여러 개 열려 있으면 url 패턴을 명시한다 — `fixtureTabId("http://127.0.0.1/basic.html")`. 기본 패턴(`http://127.0.0.1/*`)은 첫 매칭 탭을 잡는다(chrome match pattern은 포트 무시).
- **`aria-disabled` 가드 버튼**: `next-step` 등은 `disabled`가 아니라 `aria-disabled`+클릭 가드다. Playwright actionability가 안 막으므로 클릭 전 `expect(btn).not.toHaveAttribute("aria-disabled","true")` 단언 필수 — 없으면 조용한 no-op.
- **`aria-disabled="true"` 버튼을 *클릭해야* 할 때는 `click({ force: true })`**: 일부 Playwright 버전은 actionability "enabled" 검사에서 `aria-disabled="true"`를 disabled로 취급해 일반 `click()`이 타임아웃까지 대기한다(`capture-modes-layout`의 비활성 `replay-button`은 onClick이 가드 없이 설정 탭으로 `navTo`하는 진짜 동작이라 `force: true`로 클릭). 위 `next-step` 케이스처럼 클릭이 no-op이어야 하는 가드 버튼과 구분 — 그쪽은 클릭 전 부재 단언이 목적.
- **`recording-mode-*` testid는 이제 SettingsTab 단일**: revert-idle-capture-layout에서 ⚙ 다이얼로그(`RecordingSettingsDialog`)를 제거해 `RecordingSettingsCard`는 설정 탭에서만 렌더된다. `recording-mode-tab`/`-screen` testid는 단일이라 `getByRole("dialog")` 스코프 불필요(과거엔 다이얼로그와 중복돼 스코프했음). 리플레이 토글은 여전히 `id="replay-enabled"`(testid 아님 — `#replay-enabled`로 접근). 카드는 설정 탭 비활성 시 DOM에 hidden으로 존재하므로 모드 전환 단언 전 `tab-settings`+`settings-sub-issue` active를 확보한다.

- **action 종류 단언은 `data-kind` + 리터럴 키 조합으로**: 행은 `data-kind`(click/input/navigation/keypress/toggle/select)를 노출하므로 종류별 `toHaveCount`로 센다. verb 문구는 i18n(ko)이라 텍스트 의존 금지지만, keypress의 **키 조합 값**(`Escape`·`Ctrl+K`·`Enter`)은 i18n 무관 리터럴이라 `hasText`로 안전하게 구분 가능. 이중기록 제거는 "그 동작만 수행하고 click을 0건으로 단언"하는 방식이 견고(action-log-coverage spec: 버튼·앵커 미클릭 → checkbox/label/radio click이 click으로 새면 0이 깨짐). 동작은 ready 확보 **후·캡처 직전**에 몰아 trim 윈도우(30s cap) 안에 확실히 넣는다.
- **네이티브 HTML5 DnD는 합성 `DragEvent`로만 재현 — Playwright는 real native drag를 못 일으킨다**: `dragTo()`·`mouse.down→move→up`은 **pointer 이벤트만** 쏴 브라우저의 native `dragstart`/`drop`을 발화하지 않는다(레코더 포인터 경로로 떨어져 source-only가 됨). 레코더가 `isTrusted`를 안 보므로 `fixture.evaluate`로 `new DragEvent("dragstart"|"dragover"|"drop"|"dragend", {bubbles:true, cancelable:true, dataTransfer: new DataTransfer()})`를 src→zone에 순서대로 dispatch해 source+target 경로를 재현한다(`drag-action.spec`). `bubbles:true`라야 document capture 리스너에 닿는다. `draggable=true`/페이지 drop 핸들러는 현실성용일 뿐 합성 dispatch엔 불필요.
- **포인터 드래그 e2e는 `user-select:none` 핸들 위에서 — 안 그러면 텍스트선택 가드에 먹힌다**: 레코더 포인터 경로는 `getSelection().isCollapsed === false`(텍스트 선택 중)면 드래그를 기록하지 않는다(팬·텍스트선택 제외 가드). `mouse.down→move(steps)→up`으로 페이지를 끌면 경로의 텍스트가 선택돼 이 가드가 발동 → drag 0건, 대신 down/up 공통조상(body)의 **click**이 1건 기록된다(증상: 드래그 기대 위치에 큰 텍스트 묶음의 click 엔트리). fixture 드래그 핸들에 `user-select:none`(실제 dnd-kit 관습)을 줘 선택을 원천 차단하면 가드를 통과해 `drag`(source-only)로 기록된다. 포인터 이벤트 자체(down/move/up·isPrimary·button0·maxd>15·endTag≠source)는 정상이어도 이 가드 하나로 갈리므로, 포인터 드래그가 click으로 새면 **선택 발생**을 먼저 의심.
- **30s Replay 캡처(action 로그 진입로)**: 수동 video(`getUserMedia`/`tabCapture`)와 달리 Replay는 `captureVisibleTab` 폴링이라 e2e `<all_urls>`로 동작한다(`replay-action-log.spec`). 세 가지 함정: ① **활성화** — 설정 issue 서브탭 `#replay-enabled` 스위치 토글. `<all_urls>`가 required라 토글이 권한 확인 없이 즉시 켜진다(리소스 점유 opt-in). chrome.storage 영속이므로 **`afterAll`로 off 복원**(설정 오염 함정). ② **버퍼 readiness** — Replay tick은 **fixture 탭이 active(front)**일 때만 `captureVisibleTab`한다. `fixture.bringToFront()`로 front를 잡고 ≥10프레임(600ms 간격) 쌓일 때까지 대기. 이때 패널은 백그라운드 탭이라 타이머가 throttle돼 프레임이 ~1/s로 느리게 쌓이니 ready 폴링 timeout을 넉넉히(≥45s). ready=`replay-button`의 `aria-disabled` 해제. ③ **action 캡처 타이밍** — 백그라운드 레코더가 idle 중 상시 action을 버퍼하고 `capture()`가 trim 윈도우(가드밴드 1500ms, 30s cap)로 잘라 첨부한다. 동작은 bringToFront 직후 수행하면 윈도우 안에 든다. 캡처→`encodeToMp4`(WebCodecs H.264, headed Chrome 동작 확인) 후 drafting(video) 전환 → action 로그 카드(`action-log-card`)/다이얼로그(행은 `data-kind` = click/input/navigation). **다이얼로그는 모달이라 열린 채 두면 후속 탭 전환 클릭이 오버레이에 막힌다 → 단언 후 Escape로 닫는다.**
- **`captureVisibleTab` rate-limit는 spec 경계를 넘는다**: Chrome은 `captureVisibleTab`에 `MAX_CAPTURE_VISIBLE_TAB_CALLS_PER_SECOND`(~2회/초) quota를 건다. quota 초과 시 reject → 제품의 `captureAndCrop`이 `reset()`해 panel이 idle로 복귀(drafting 안 뜸). 이 quota는 **확장 전역**이라 한 spec의 캡처가 같은 run의 후행 spec까지 영향을 준다 — 알파벳 순서상 `action-log-coverage`·(이전) `activetab-broad-permission`의 캡처 폴링이 quota를 소진하면 뒤따르는 `capture.spec`이 간헐 red(단독 실행·repeat-each로는 재현 안 됨 — full-suite 부하에서만). 대응 두 가지: ① **캡처를 폴링하는 헬퍼는 버스트 금지** — `expect.poll`에 `intervals: [1000,...]`로 1초 간격 spaced 재시도 + throw를 ""로 흡수(`activetab-broad-permission`의 `captureVisibleTab`/`CAPTURE_POLL`). ② **캡처→drafting을 검증하는 spec은 quota 회복을 기다리는 재시도** — `captureUntilDrafting`(capture.spec)가 트리거(mode+선택)를 drafting 진입까지 1초+ 간격 `toPass`로 재시도(이미 drafting이면 재트리거 스킵). timeout 늘리기가 아니라 **간격을 quota 주기에 맞추는** 게 핵심.
- **패널 재오픈 후 탭은 비활성 — drafting-panel이 hidden**: 세션 복원 검증으로 `panel.close()` → `openPanel(tabId)` 하면 재오픈 패널의 기본 활성 탭이 debug가 아니라 drafting-panel이 `data-[state=inactive]:hidden`으로 숨는다(DOM엔 존재). 복원 확인 전 **`enterDebug(panel)`로 debug 탭을 다시 활성화**해야 visible 단언이 통과한다(`attachments.spec`). 또 닫기 전 session snapshot debounce(300ms)가 flush되도록 `waitForTimeout(400)` 후 닫는다.
- **`setInputFiles`는 hidden input에 직접 set + 비동기 반영**: 파일 첨부 `<input type=file>`은 `hidden`이지만 Playwright `setInputFiles`는 actionability를 우회해 그대로 set된다(디스크 파일 불필요 — `{name, mimeType, buffer}` 인라인). 단 `addAttachments`가 `saveAttachmentBlob`(IndexedDB) await 후 메타를 push하므로 추가 결과는 **`toHaveCount` polling**으로 기다린다(동기 단언 금지). 카운터는 버튼 라벨에 `n/10`으로 박혀 i18n 무관 숫자라 `toContainText("2/10")`로 안전하게 단언.
- **다운로드 버튼은 `waitForEvent("download")`로 판정 — 확장 패널(일반 Page)에서도 발화**: 미디어·로그 다운로드는 `URL.createObjectURL(blob)` + `<a download>` 클릭이라 `panel.waitForEvent("download")`가 잡는다(extension page도 동일, `acceptDownloads` 기본 true). `Promise.all([waitForEvent("download"), btn.click()])`로 경합 없이 받고 `suggestedFilename()`만 단언(파일 내용=바이너리/픽셀이라 판정 불가). 파일명은 i18n 무관 리터럴(`screenshot.webp`/`logs.html`/`recording.mp4`)이라 안전. data URL이 아닌 **blob URL**이어야 이벤트가 안정적으로 뜬다(제품 `downloadCapture`는 전부 blob URL — 스크린샷도 `dataUrlToBlob` 경유).

- **log-viewer는 확장 없이 standalone HTML을 `setContent`로 연다**: `dist-log-viewer/index.html`은 `viteSingleFile` 빌드라 css/js가 한 장에 inline. `#__BUGSHOT_DATA__` placeholder script 태그에 합성 `LogViewerData`를 **평문 JSON**으로 박아 넣으면 `main.tsx`가 평문 경로(type이 `application/gzip-base64`가 아님)로 파싱한다 → gzip 인코딩 불필요. 주입 JSON은 `</script>` 깨짐 방지로 `<`를 `<`로 escape(`openViewer` 헬퍼). dist를 읽으므로 **src의 testid를 바꾸면 `build:log-viewer` 재빌드 후** 실행. video/screenshot 미지정 시 단일 패널, 지정 시 좌우 분할(ResizablePanel) — 둘 다 `logview-tab-*`로 탭 접근.
- **log-viewer i18n은 `navigator.language` 직접 판정 — Playwright `locale`로 결정적**: 사이드패널 앱과 달리(locale 비결정 함정) `src/log-viewer/i18n.ts`는 `navigator.language.startsWith("ko")`로 dict를 고르므로 `test.use({ locale: "ko-KR" | "en-US" })`로 ko/en을 확정한다. 그래서 **이 spec에선 i18n 라벨 텍스트 단언이 정당**(라벨 정확성이 SUT). 라벨이 키 문자열로 새는 회귀는 `not.toContainText("Log.filter")` + 정확 라벨 `toHaveText`로 잡는다.

- **logview spec은 `@/` import 금지 — `../../src` 상대경로 + transitive `@/` 회피**: `tsconfig.e2e.json`엔 `paths`(`@/`)가 없다. e2e에서 src 타입을 쓸 땐 상대경로로 import하되, **그 파일이 다시 `@/`를 import하면 tsc가 깨진다**(`logOrigin.ts`→`@/lib/session-keys`). 타입(`src/types/*`)은 자체 완결이라 안전하지만, 헬퍼 함수가 필요하면 로직을 spec에 인라인한다(예: `originKey` = `new URL(url).origin`). `pnpm typecheck`(tsc -b)가 e2e도 포함하므로 위반 시 빌드가 막힌다.
- **uncaught error 스택에 page URL을 넣으려면 정적 인라인 스크립트가 필요**: 콘솔 stack/args linkify를 검증하려면 캡처된 에러의 stack·`ErrorEvent.filename`에 `http://` URL이 있어야 한다. `console.error(...)`의 stack은 `captureStack()`(=레코더 call-site=evaluate eval 프레임)이라 깨끗한 http URL이 없고, **동적 주입**(`createElement`+`textContent`) 인라인 스크립트의 throw도 `filename`이 비어 args에 URL이 안 들어간다(관측: args=`Uncaught Uncaught Error: ...`, loc 없음). 해결: fixture **정적 HTML**(`console-error.html`)의 인라인 스크립트 함수를 `setTimeout`으로 async throw → uncaught → `formatErrorEvent`가 `filename`(page URL)을 args에, `cleanStack(error.stack)`이 page 프레임 URL을 stack에 담는다(`console-linkify.spec`).
- **target=_blank 링크 클릭 검증은 capture-phase preventDefault로 네비 억제**: 콘솔 헤더 URL 링크는 `InlineLink`라 `target="_blank"` — Playwright `click()`이 새 탭(popup)을 띄운다. stopPropagation(행 토글 차단)만 검증하려면 네비게이션은 막아야 깔끔하다. `panel.evaluate`로 document에 **capture 단계** click 리스너를 달아 `a` 조상이면 `preventDefault()`(네비만 차단). React onClick(bubble)은 그대로 실행되어 링크의 `stopPropagation`이 행 토글을 막는지 단언할 수 있다(`console-linkify.spec`). 클릭 후 `console-stack` 부재로 "안 펼쳐짐" 판정.
- **SSRF 가드가 loopback cross-origin sheet를 차단 — cross-origin 스타일 보강 양성 경로는 e2e 불가**: cross-origin-styles 보강은 background가 외부 CSS를 fetch하는데, `isFetchableSheetUrl` SSRF 가드가 `localhost`·`127.x`·사설/link-local IP를 차단한다(page-controlled href로 내부망 SSRF 방지). 그런데 e2e fixture의 cross-origin 분리 트릭은 **127.0.0.1 top + localhost sheet**(같은 loopback 서버, 호스트명만 다름)라 — 가드가 localhost sheet fetch를 정상 차단 → 보강이 안 일어난다. 즉 e2e 환경에선 보강이 **항상 inert**고, cross-origin 페이지는 가드 차단 → computed fallback(`style-cross-origin-section.spec`이 이걸 green으로 락). 양성 보강을 e2e로 타려면 가드를 통과하는 공개 호스트가 필요한데 loopback 전용 서버엔 없음 → 수동 잔여. 가드 자체는 `ssrf-guard.test.ts` 단위로 직접 커버하니 e2e에서 가드를 우회하려 들지 말 것(보안 약화).
