# e2e 함정 (gotchas — 실전에서 밟은 것 누적)

새 spec 쓰기 전에 훑어 같은 함정을 반복하지 않는다. 개요·실행법·헬퍼는 [README.md](./README.md), 커버리지·수동 잔여는 [COVERAGE.md](./COVERAGE.md).

- **30s Replay 캡처 spec은 제거됨**: replay-action-log·replay-trim·replay-trim-logs·action-log-coverage·drag-action은 captureVisibleTab cold-start(새 컨텍스트 첫 캡처가 drafting을 못 만들고 idle 리셋) + extension-global quota로 환경 flaky가 심해 게이트를 신뢰 불가하게 만들어 전부 삭제했다. 캡처를 진입로로 쓰는 새 spec은 같은 함정을 밟으니, 트림·액션·드래그 로직은 단위 테스트로, 캡처 경로는 수동으로 커버한다.
- **spec 간 탭 누수**: worker fixture(persistent context)를 모든 spec이 공유한다. `beforeAll`로 연 탭/패널은 **반드시 `afterAll`로 닫는다**. 안 닫으면 후행 spec의 `fixtureTabId()`가 잔여 탭을 잡아 **실행 순서에 따라 실패**한다(파일명 알파벳 순 실행 — rename으로 순서가 바뀌면 드러난다).
- **`fixtureTabId` 모호성**: fixture 탭이 여러 개 열려 있으면 url 패턴을 명시한다 — `fixtureTabId("http://127.0.0.1/basic.html")`. 기본 패턴(`http://127.0.0.1/*`)은 첫 매칭 탭을 잡는다(chrome match pattern은 포트 무시).
- **`aria-disabled` 가드 버튼**: `next-step` 등은 `disabled`가 아니라 `aria-disabled`+클릭 가드다. Playwright actionability가 안 막으므로 클릭 전 `expect(btn).not.toHaveAttribute("aria-disabled","true")` 단언 필수 — 없으면 조용한 no-op.
- **`aria-disabled="true"` 버튼을 *클릭해야* 할 때는 `click({ force: true })`**: 일부 Playwright 버전은 actionability "enabled" 검사에서 `aria-disabled="true"`를 disabled로 취급해 일반 `click()`이 타임아웃까지 대기한다(`capture-modes-layout`의 비활성 `replay-button`은 onClick이 가드 없이 설정 탭으로 `navTo`하는 진짜 동작이라 `force: true`로 클릭). 위 `next-step` 케이스처럼 클릭이 no-op이어야 하는 가드 버튼과 구분 — 그쪽은 클릭 전 부재 단언이 목적.
- **`recording-mode-*` testid는 이제 SettingsTab 단일**: revert-idle-capture-layout에서 ⚙ 다이얼로그(`RecordingSettingsDialog`)를 제거해 `RecordingSettingsCard`는 설정 탭에서만 렌더된다. `recording-mode-tab`/`-screen` testid는 단일이라 `getByRole("dialog")` 스코프 불필요(과거엔 다이얼로그와 중복돼 스코프했음). 리플레이 토글은 여전히 `id="replay-enabled"`(testid 아님 — `#replay-enabled`로 접근). 카드는 설정 탭 비활성 시 DOM에 hidden으로 존재하므로 모드 전환 단언 전 `tab-settings`+`settings-sub-issue` active를 확보한다.
- **Konva Stage 드래그는 `panel.mouse` + 캔버스 boundingBox로**: 주석 오버레이의 Stage는 `natW×natH` 캔버스를 CSS `transform: scale()`로 축소 표시하지만 Konva `getPointerPosition`이 transform을 보정하므로 **화면 좌표 드래그를 그대로 흘려도** 도형 좌표가 맞는다. `overlay.locator("canvas").first().boundingBox()`로 표시 영역을 잡아 그 안에서 `mouse.move→down→move(steps)→up`. rect/ellipse는 **width·height 둘 다 0이 아니어야** 커밋되니 대각선 드래그(예: 0.3→0.7 지점)를 쓴다(축평행이면 `isEmptyShape`로 폐기 — `shapes.test.ts` 참고). 도형 자체는 canvas 픽셀이라 단언 불가 → done 버튼 활성/img src 전이 같은 **외부 신호**로만 판정. `annotation-done`은 `disabled`(aria 아님)라 도형 없으면 actionability가 클릭을 막는다 → 드래그로 커밋 후 `toBeEnabled()` 확인하고 클릭.
- **접힌 collapsible Section은 자식이 DOM에서 제거된다 — 펼쳐야 prop 필드가 잡힌다**: `Section`은 `{(!collapsible || open) && children}`라 접힌 섹션은 내부 prop 행이 아예 DOM에 없다(hidden 아님). `sectionDefaultOpen`이 false면 기본 접힘이라, **그 섹션의 prop이 전부 기본값인 요소**(예: position류 미설정 `#title`의 Position 섹션)를 선택하면 `propRow`/`typeStyleValue`가 라벨을 못 찾아 timeout. 토글 testid(`section-position-toggle` = `${testId}-toggle`)로 `ensureSectionOpen`을 먼저 호출해 펼친다. 섹션 testid는 `Section`의 `testId` prop으로 부착(현재 `section-position`만 — 시나리오 요구분).
- **`position:relative` 요소의 inset 미설정 computed는 `auto`가 아니라 used value `0px`**: `getComputedStyle(el).top`은 positioned 요소면 used value를 돌려줘 `top:auto`라도 `0px`로 보인다. inset 변경 제거(키 delete) 후 라이브 복원을 `toHaveCSS`로 검증하려면 `"auto"`가 아니라 **`"0px"`로 단언**한다(12px→0px). 변경이 실제로 제거됐는지는 변경 다이얼로그 `[data-prop="top"]` 행 부재로 별도 확인(`style-position-section.spec`). static 요소면 `top`은 `auto` 그대로.
- **Radix 팝오버·cmdk**: Escape가 중첩 팝오버에서 간헐 무시된다 → `closeAllPopovers`(Escape + outside-click 폴백)를 쓴다.
- **picker hover 타이밍**: hover 반영 전 클릭하면 빗나간다 → `pickElement`가 double rAF 후 클릭(내장).
- **sandbox iframe에도 content script는 주입된다 — 거부 게이트 테스트는 srcdoc으로**: `sandbox`(allow-scripts 유무 무관)는 페이지 스크립트만 막고 ISOLATED world content script는 그대로 주입된다(probe 실측 — `sandbox=""` http-src 프레임도 PING-OK). 즉 sandbox iframe은 일반 1-depth처럼 picker 선택이 *된다*. "미등록 iframe 거부" 경로를 재현하려면 주입 자체가 안 되는 프레임 — `srcdoc`(about:srcdoc, `<all_urls>` 미매치)·about:blank — 을 쓴다(`iframe-nested.html`의 `#inert`).
- **iframe 내부 요소 pick은 `pickElement`의 `{ frame }` 옵션**: frameLocator 경유 boundingBox는 메인 프레임 뷰포트 기준이라 그 좌표로 mouse.click하면 브라우저가 프레임 로컬 좌표로 라우팅한다. 첫 mousemove가 top blocker 핸드오프(등록 iframe → pointerEvents:none)를 트리거하고 클릭이 안쪽 picker에 닿는다. 거부 픽(중첩·미주입)은 다이얼로그가 뜰 때까지 클릭을 재시도(`pickUntilUnsupported` 패턴) — picker.start 직후 announce 등록이 클릭보다 늦는 창의 유실 클릭을 흡수한다(통과한 클릭은 fixture 페이지라 무해).
- **repick 직후 재선택은 단발 클릭이 유실된다(재arm 레이스) → `pickElement`가 repick 노출까지 재클릭**: repick 클릭 시 panel은 즉시 "repick 숨김"(버퍼링 상태)으로 가지만 content script picker의 재arm(start 메시지)은 비동기라, 그 창에서 떨어진 단발 클릭이 picker에 안 닿아 "Select an element"에 머문다(`buffered-reselect-edit` 간헐 red의 원인 — 단독·풀스위트 무관하게 타이밍에 따라 재현). `pickElement`는 기본으로 repick이 보일 때까지 hover+click을 `toPass`로 재시도해 흡수한다(이미 선택됐으면 picker가 idle이라 추가 클릭 무해·idempotent). **단, repick이 안 뜨는 픽은 이 재시도가 15s timeout으로 죽으므로 `{ expectSelection:false }`** — element-shot 캡처(`capture.spec`, 곧장 drafting)·iframe 미지원(`picker-guard.spec`, unsupported 다이얼로그)이 해당. element-shot은 `captureUntilDrafting`이 전체 트리거(mode+pick)를 재시도하므로 유실 클릭이 거기서 복구된다.
- **repick 후 재선택**: repick 클릭 직후 바로 pick하지 말 것. 버퍼 스냅샷 캡처 완료(=`repick` 버튼 소실)를 기다린 뒤 pick한다.
- **로그 sync 타이밍**: 패널 마운트 시 레코더가 자동 활성화되지만 완료 신호가 없다. 로그 발생 + sync 주기(1500ms) 대기를 `expect(...).toPass()` polling으로 반복해 첫 캡처를 기다린다. **로그 탭은 `recording`/`drafting`/`previewing`/`done` phase(`logTabsLocked`)에서 비활성** — 로그 테스트는 idle에서. (`done`은 제출 후라 e2e 미도달 — OAuth 전제, 수동 잔여)
- **fresh 프로필 integrations 자동 전환**: 연동 0개면 integrations 탭으로 자동 전환되는 effect와 race가 난다. `enterDebug`가 `tab-debug` active 폴링으로 흡수한다 — 디버그 탭 진입은 이 헬퍼를 거친다. **다른 메인 탭(settings 등) 진입도 동일하게 클릭+active 폴링** 필요(settings-sections.spec의 `setSectionEnabled` 참고).
- **설정 영속 오염**: settings-ui-store는 chrome.storage에 영속돼 worker run 내 후행 spec까지 새어간다. 설정을 바꾸는 spec은 **`finally`로 기본값 복원**한다.
- **셀렉터**: `data-testid` 우선. 섹션 제목 등 i18n 텍스트 의존 금지. CSS prop 라벨(`color`/`padding`)만 하드코딩 허용. testid가 없으면 src에 **속성 추가만**(로직·구조 변경 금지) 후 `build:e2e` 재빌드. (예: 콘솔 엔트리 행은 `data-level`(log/info/warn/error/debug)을 노출 — 탭 필터 의존 없이 `[data-entry-id][data-level="error"]`로 레벨 단언.)
- **bg 패널 종료/유지(deactivatePanelIfCrossOrigin) 판정**: e2e 패널은 실제 side panel이 아니라 일반 Page라 `setOptions(enabled:false)`로 닫히지 않는다 → `waitForEvent("close")`로 판정 불가. 대신 bg가 유지하는 `sidePanel:activated`(number[]) / `editor:${tabId}` 세션 키를 SW `chrome.storage.session`으로 읽어 deactivate/keep을 판정한다(`activetab-broad-permission.spec`). 또한 이 분기는 **activated 탭에서만** 동작하는데(미활성 early-return), activated set은 액션 아이콘 클릭(activateTab)으로만 채워지고 Playwright는 확장 액션을 클릭할 수 없다 → SW로 직접 seed해 전제를 만든다. 탭 닫힘 시 `onRemoved`가 seed 키를 자동 정리하므로 후행 spec 누수 없음.
- **full-suite headed 실행 시 Chrome "여러 파일 다운로드 허용?" 확인창이 다운로드 spec을 간헐 차단**: `download-buttons.spec`가 한 컨텍스트에서 다중 다운로드를 트리거하면 Chrome이 "이 사이트에서 파일 여러 개를 다운로드하려고 합니다" 네이티브 확인창을 띄워 후속 `waitForEvent("download")`가 막힐 수 있다(full-suite에서만, **단독 실행은 2/2 green**). 코드 결함이 아니라 환경 아티팩트 — `download-buttons` 단독 재실행으로 판정한다. (항구 대응이 필요하면 컨텍스트에 `--disable-features` 또는 download 정책 옵션 검토 — 현재는 단독 재실행으로 갈음.)
- **`captureVisibleTab` rate-limit는 spec 경계를 넘는다**: Chrome은 `captureVisibleTab`에 `MAX_CAPTURE_VISIBLE_TAB_CALLS_PER_SECOND`(~2회/초) quota를 건다. quota 초과 시 reject → 제품의 `captureAndCrop`이 `reset()`해 panel이 idle로 복귀(drafting 안 뜸). 이 quota는 **확장 전역**이라 한 spec의 캡처가 같은 run의 후행 spec까지 영향을 준다. 대응 두 가지: ① **캡처를 폴링하는 헬퍼는 버스트 금지** — `expect.poll`에 `intervals: [1000,...]`로 1초 간격 spaced 재시도 + throw를 ""로 흡수(`activetab-broad-permission`의 `captureVisibleTab`/`CAPTURE_POLL`). ② **캡처→drafting을 검증하는 spec은 quota 회복을 기다리는 재시도** — `captureUntilDrafting`(capture.spec)가 트리거(mode+선택)를 drafting 진입까지 1초+ 간격 `toPass`로 재시도(이미 drafting이면 재트리거 스킵). timeout 늘리기가 아니라 **간격을 quota 주기에 맞추는** 게 핵심.
- **패널 재오픈 후 탭은 비활성 — drafting-panel이 hidden**: 세션 복원 검증으로 `panel.close()` → `openPanel(tabId)` 하면 재오픈 패널의 기본 활성 탭이 debug가 아니라 drafting-panel이 `data-[state=inactive]:hidden`으로 숨는다(DOM엔 존재). 복원 확인 전 **`enterDebug(panel)`로 debug 탭을 다시 활성화**해야 visible 단언이 통과한다(`attachments.spec`). 또 닫기 전 session snapshot debounce(300ms)가 flush되도록 `waitForTimeout(400)` 후 닫는다.
- **미디어가 captureFiles에 들어가야 하는 제출 경로는 IndexedDB blob을 직접 seed**: 제출/승격의 미디어(이미지·비디오·첨부)는 issue record가 아니라 IndexedDB `bugshot-video` DB(store `images`/`blobs`/`attachments` …, image key `${id}:${slot}`)에 있다. storage seed한 이슈는 record만 있어 `buildCtxForSubmit`의 `captureFiles.images`가 비므로, 이미지를 요구하는 경로(예: 승격 미디어 업로드 가드)는 안 탄다. `panel.evaluate`로 `indexedDB.open("bugshot-video", 7)` + `onupgradeneeded`에 **7개 store 전부**(blobs/images/networkLogs/consoleLogs/actionLogs/inlineImages/attachments) `contains` 가드 생성(앱 `openDb`와 동일 스키마라야 버전 충돌 없음) 후 `images`에 `new Blob([...],{type:"image/webp"})`를 `${id}:before`로 put. 이슈는 `captureMode:"screenshot"`+`snapshot.before:true`라야 `buildCaptureFiles`가 `screenshot.webp` 이미지를 만든다(`slack-promote-media-guard.spec`). key가 이슈 id로 namespace돼 다른 spec과 안 부딪혀 cleanup 불요(orphan은 pruneOrphanBlobs가 정리).
- **`setInputFiles`는 hidden input에 직접 set + 비동기 반영**: 파일 첨부 `<input type=file>`은 `hidden`이지만 Playwright `setInputFiles`는 actionability를 우회해 그대로 set된다(디스크 파일 불필요 — `{name, mimeType, buffer}` 인라인). 단 `addAttachments`가 `saveAttachmentBlob`(IndexedDB) await 후 메타를 push하므로 추가 결과는 **`toHaveCount` polling**으로 기다린다(동기 단언 금지). 카운터는 버튼 라벨에 `n/10`으로 박혀 i18n 무관 숫자라 `toContainText("2/10")`로 안전하게 단언.
- **다운로드 버튼은 `waitForEvent("download")`로 판정 — 확장 패널(일반 Page)에서도 발화**: 미디어·로그 다운로드는 `URL.createObjectURL(blob)` + `<a download>` 클릭이라 `panel.waitForEvent("download")`가 잡는다(extension page도 동일, `acceptDownloads` 기본 true). `Promise.all([waitForEvent("download"), btn.click()])`로 경합 없이 받고 `suggestedFilename()`만 단언(파일 내용=바이너리/픽셀이라 판정 불가). 파일명은 i18n 무관 리터럴(`screenshot.webp`/`logs.html`/`recording.mp4`)이라 안전. data URL이 아닌 **blob URL**이어야 이벤트가 안정적으로 뜬다(제품 `downloadCapture`는 전부 blob URL — 스크린샷도 `dataUrlToBlob` 경유).
- **로그 섹션 다운로드 테스트는 캡처 전 issue 서브탭 복귀 필수**: drafting 로그 카드는 store에 console/network 로그가 있어야 노출되므로 idle에서 먼저 로그를 적재한다(log-capture 패턴 — `subtab-console`에서 `[data-entry-id]` polling). 그 다음 스크린샷 캡처를 하려면 **`subtab-issue` 클릭으로 캡처 진입 화면으로 돌아와야** `mode-screenshot`이 보인다(console 서브탭엔 없음). `subtab-issue` testid는 이 목적에 추가됨.

- **log-viewer는 확장 없이 standalone HTML을 `setContent`로 연다**: `dist-log-viewer/index.html`은 `viteSingleFile` 빌드라 css/js가 한 장에 inline. `#__BUGSHOT_DATA__` placeholder script 태그에 합성 `LogViewerData`를 **평문 JSON**으로 박아 넣으면 `main.tsx`가 평문 경로(type이 `application/gzip-base64`가 아님)로 파싱한다 → gzip 인코딩 불필요. 주입 JSON은 `</script>` 깨짐 방지로 `<`를 `<`로 escape(`openViewer` 헬퍼). dist를 읽으므로 **src의 testid를 바꾸면 `build:log-viewer` 재빌드 후** 실행. video/screenshot 미지정 시 단일 패널, 지정 시 좌우 분할(ResizablePanel) — 둘 다 `logview-tab-*`로 탭 접근.
- **log-viewer i18n은 `navigator.language` 직접 판정 — Playwright `locale`로 결정적**: 사이드패널 앱과 달리(locale 비결정 함정) `src/log-viewer/i18n.ts`는 `navigator.language.startsWith("ko")`로 dict를 고르므로 `test.use({ locale: "ko-KR" | "en-US" })`로 ko/en을 확정한다. 그래서 **이 spec에선 i18n 라벨 텍스트 단언이 정당**(라벨 정확성이 SUT). 라벨이 키 문자열로 새는 회귀는 `not.toContainText("Log.filter")` + 정확 라벨 `toHaveText`로 잡는다.
- **standalone 페이지엔 `addInitScript`가 안 먹는다 — `setContent` 후 `evaluate`로 stub**: log-viewer copy(`copyReport`)는 `navigator.clipboard.write`→실패 시 `writeText` 폴백이라, 권한 없는 테스트에선 둘 다 실패해 `setCopied`에 도달 못 한다(Check 아이콘 안 뜸). `addInitScript`로 stub하면 setContent 페이지엔 적용 안 돼 미해결 → `openViewer` **후** `page.evaluate`로 `navigator.clipboard`를 덮는다(핸들러가 클릭 시점에 읽으므로 마운트 후 덮어도 반영). copied 전환은 i18n 라벨 대신 `svg.lucide-check` 아이콘으로 판정. 다운로드는 blob `<a download>`라 `setContent` 페이지에서도 `waitForEvent("download")`로 잡힌다(파일명은 i18n 무관 리터럴).
- **logview spec은 `@/` import 금지 — `../../src` 상대경로 + transitive `@/` 회피**: `tsconfig.e2e.json`엔 `paths`(`@/`)가 없다. e2e에서 src 타입을 쓸 땐 상대경로로 import하되, **그 파일이 다시 `@/`를 import하면 tsc가 깨진다**(`logOrigin.ts`→`@/lib/session-keys`). 타입(`src/types/*`)은 자체 완결이라 안전하지만, 헬퍼 함수가 필요하면 로직을 spec에 인라인한다(예: `originKey` = `new URL(url).origin`). `pnpm typecheck`(tsc -b)가 e2e도 포함하므로 위반 시 빌드가 막힌다.
- **uncaught error 스택에 page URL을 넣으려면 정적 인라인 스크립트가 필요**: 콘솔 stack/args linkify를 검증하려면 캡처된 에러의 stack·`ErrorEvent.filename`에 `http://` URL이 있어야 한다. `console.error(...)`의 stack은 `captureStack()`(=레코더 call-site=evaluate eval 프레임)이라 깨끗한 http URL이 없고, **동적 주입**(`createElement`+`textContent`) 인라인 스크립트의 throw도 `filename`이 비어 args에 URL이 안 들어간다(관측: args=`Uncaught Uncaught Error: ...`, loc 없음). 해결: fixture **정적 HTML**(`console-error.html`)의 인라인 스크립트 함수를 `setTimeout`으로 async throw → uncaught → `formatErrorEvent`가 `filename`(page URL)을 args에, `cleanStack(error.stack)`이 page 프레임 URL을 stack에 담는다(`console-linkify.spec`).
- **target=_blank 링크 클릭 검증은 capture-phase preventDefault로 네비 억제**: 콘솔 헤더 URL 링크는 `InlineLink`라 `target="_blank"` — Playwright `click()`이 새 탭(popup)을 띄운다. stopPropagation(행 토글 차단)만 검증하려면 네비게이션은 막아야 깔끔하다. `panel.evaluate`로 document에 **capture 단계** click 리스너를 달아 `a` 조상이면 `preventDefault()`(네비만 차단). React onClick(bubble)은 그대로 실행되어 링크의 `stopPropagation`이 행 토글을 막는지 단언할 수 있다(`console-linkify.spec`). 클릭 후 `console-stack` 부재로 "안 펼쳐짐" 판정.
- **content script 오버레이는 SW `chrome.tabs.sendMessage`로 직구동 — 패널 불필요**: 녹화 그리기 오버레이(`annotation.*`)처럼 picker 엔트리 내부 모듈이 처리하는 메시지는 패널을 열지 않고도 검증한다. picker(`content_scripts[0]`, all_frames)가 fixture 페이지 로드 시 자동 주입돼 `handlePickerMessage` 리스너를 등록하므로, `ext.context.serviceWorkers()[0]`를 잡아 `sw.evaluate(([t,m])=>chrome.tabs.sendMessage(t,m), [tabId,msg])`로 top frame에 직접 보낸다. picker가 `sendResponse({ok:true})`로 동기 응답하므로 `await sendMessage`가 핸들링 완료를 보장(별도 대기 불요). **open shadow** host(`attachShadow({mode:"open"})`)라 `document.getElementById(HOST_ID).shadowRoot`를 fixture `evaluate`에서 그대로 읽어 내부 DOM(`svg g` 획 수·`.blocker` 클래스)을 판정할 수 있다(`recording-annotation.spec`). closed shadow였다면 불가.
- **shadow blocker 드래그·pass-through는 `fixture.mouse`로 — pointer 이벤트 합성됨**: 오버레이 blocker(shadow 내부, `pointer-events:auto` when armed)에 그리려면 `fixture.mouse.move→down→move(steps)→up`. CDP `Input.dispatchMouseEvent`가 trusted `pointerdown/move/up`을 함께 합성하므로 blocker의 `pointerdown` + window capture `pointermove/up`이 발화한다(`e.button===0` 기본 좌클릭 충족). 펜 OFF면 blocker `pointer-events:none`이라 같은 좌표 클릭이 **페이지로 통과** → fixture에 `click` 리스너를 달아 pass-through를 판정. **3초 페이드로 획이 사라지므로** 획 수 단언은 커밋 직후(수 초 내)에 하거나, 이전 테스트 획과 격리하려면 `hide`+`show`로 오버레이를 새로 마운트해 획 0에서 시작한다(fade race 회피 — `recording-annotation.spec`의 펜 OFF 테스트).
- **SSRF 가드가 loopback cross-origin sheet를 차단 — cross-origin 스타일 보강 양성 경로는 e2e 불가**: cross-origin-styles 보강은 background가 외부 CSS를 fetch하는데, `isFetchableSheetUrl` SSRF 가드가 `localhost`·`127.x`·사설/link-local IP를 차단한다(page-controlled href로 내부망 SSRF 방지). 그런데 e2e fixture의 cross-origin 분리 트릭은 **127.0.0.1 top + localhost sheet**(같은 loopback 서버, 호스트명만 다름)라 — 가드가 localhost sheet fetch를 정상 차단 → 보강이 안 일어난다. 즉 e2e 환경에선 보강이 **항상 inert**고, cross-origin 페이지는 가드 차단 → computed fallback(`style-cross-origin-section.spec`이 이걸 green으로 락). 양성 보강을 e2e로 타려면 가드를 통과하는 공개 호스트가 필요한데 loopback 전용 서버엔 없음 → 수동 잔여. 가드 자체는 `ssrf-guard.test.ts` 단위로 직접 커버하니 e2e에서 가드를 우회하려 들지 말 것(보안 약화).
